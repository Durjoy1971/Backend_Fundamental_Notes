# Validations & Transformations: A Complete Guide

> **Prerequisites:** Client-server architecture, HTTP, REST APIs, JSON.

---

## 1. Backend Architecture: Three Layers

A well-structured backend has three layers:

| Layer | Responsibility | Example |
|-------|--------------|---------|
| **Controller** | HTTP logic: receives requests, returns responses, validates data | Receives JSON, returns 200 or 400 |
| **Service** | Business logic: what the API actually does | Call repo, send email, webhook |
| **Repository** | Database interaction: queries, inserts, updates | `INSERT INTO books ...` |

**Why separate Controller and Service?**
- Controller handles HTTP-specific concerns (status codes, request/response formats, validation)
- Service handles business rules (what operations to perform)
- This keeps HTTP concerns isolated from business logic

---

## 2. The Validation & Transformation Pipeline

**Where it happens:** Right at the entry point, after route matching and before any business logic.

```
Client Request
      ↓
Route Matching
      ↓
VALIDATION & TRANSFORMATION  <- CRITICAL: happens FIRST
      ↓
Controller → Service → Repository → Response
```

**Why at the entry point?**

Without validation, unexpected data crashes your server:

```json
// Client sends:
{ "name": 0 }

// Server expects string for database TEXT column
// INSERT fails → 500 Internal Server Error
```

With validation:
```json
// Validation catches it immediately
// Returns 400 Bad Request
// Service and Database are never touched
```

> **Golden Rule:** Validate everything from the client before doing anything significant.

## 3. Types of Validation

### 3.1 Syntactic Validation
Does the data follow the expected structure?

| Field | Check | Example |
|-------|-------|---------|
| Email | `user@domain.tld` | `test@gmail.com` ✅ `test.com` ❌ |
| Phone | Country code + digits | `+91-9876543210` ✅ `12345` ❌ |
| Date | `YYYY-MM-DD` format | `2025-11-05` ✅ `05/11/2025` ❌ |
| URL | Valid URL structure | `https://example.com` ✅ |

---

### 3.2 Semantic Validation
Does the data make logical sense?

| Field | Check | Example |
|-------|-------|---------|
| Date of Birth | Cannot be in future | `2026-01-01` ❌ |
| Age | Within human range | `430` ❌ `43` ✅ |
| Price | Must be positive | `$-10` ❌ |

---

### 3.3 Type Validation
Is the data the expected type?

| Expected | Received | Result |
|----------|----------|--------|
| `string` | `123` (number) | ❌ Expected string, got number |
| `number` | `"10"` (string) | ❌ Expected number, got string |
| `boolean` | `"true"` (string) | ❌ Expected boolean, got string |
| `array` | `"[1,2]"` (string) | ❌ Expected array, got string |
| `array[string]` | `[1, 2]` (numbers) | ❌ Array element 0: expected string, got number |

---

## 4. Complex Validations

### Cross-Field Validation
Two fields must relate to each other:

```json
{
  "password": "secret123",
  "passwordConfirmation": "different"
}
// Error: passwords don't match
```

### Conditional Validation
A field is required only when another field has a specific value:

```json
{
  "married": true
}
// Error: partner name is required when married is true

{
  "married": true,
  "partner": "Alice"
}
// ✅ Success
```

## 5. Transformations

**Transformation = converting data into the format your system needs.**

Validations and transformations usually happen in the same pipeline.

### Common Transformations

| Input | Transformation | Output |
|-------|---------------|--------|
| `User@Email.COM` | Lowercase | `user@email.com` |
| `9876543210` | Add country code | `+91-9876543210` |
| `2025-11-05` | Reformat date | `05/11/2025` |
| `"2"` (query param) | Cast to number | `2` |
| `"20"` (query param) | Cast to number | `20` |

### Query Parameter Example

Query parameters are **always strings** when they reach your server:

```
GET /api/bookmarks?page=2&limit=20
```

Your server receives:
```json
{ "page": "2", "limit": "20" }
```

Your validation expects numbers:
```
page: number, must be > 0 and < 500
limit: number, must be > 0 and < 10000
```

**Problem:** Validation fails because `"2"` is a string, not a number.

**Solution:** Transform first, then validate:
```
1. Cast "2" → 2 (number)
2. Cast "20" → 20 (number)
3. Validate: 2 > 0 ✅, 20 > 0 ✅
```

> **Pipeline order:** Transform → Validate → Execute Business Logic

## 6. Frontend vs Backend Validation

**A common mistake:** Thinking frontend validation replaces backend validation.

### Frontend Validation
- **Purpose:** User Experience (UX)
- **Gives immediate feedback** before submitting
- **Not for security** — can be bypassed

### Backend Validation
- **Purpose:** Security and Data Integrity
- **Mandatory** — cannot be bypassed
- **Protects against malicious clients** (Postman, Insomnia, scripted attacks)

### Why You Need Both

```
User fills form → Frontend validates → Looks good → Submit
                                              ↓
                                    Attacker bypasses frontend
                                    Sends raw HTTP request
                                              ↓
                                    Backend validation catches it
                                    Returns 400 Bad Request
```

> **Critical Rule:** Never rely on frontend validation for security. Backend validation is mandatory. Frontend validation is a UX enhancement.

## 7. Key Takeaways

### Validation Types
- **Syntactic:** Does it follow the right format? (email, phone, date)
- **Semantic:** Does it make logical sense? (age ≤ 120, DOB not in future)
- **Type:** Is it the right data type? (string, number, boolean, array)

### Validation Pipeline
1. Route match → Controller method called
2. **Transform** data (cast types, normalize)
3. **Validate** data (syntactic, semantic, type checks)
4. If valid → proceed to Service layer
5. If invalid → return 400 Bad Request immediately

### Transformations
- Query parameters are always strings — cast before validating
- Normalize emails to lowercase, format phone numbers, reformat dates
- Transformations and validations live in the same pipeline

### Security
- Frontend validation = UX (optional, nice to have)
- Backend validation = Security & Data Integrity (mandatory)
- Design backend validation without assuming any frontend exists

## 8. Glossary

| Term | Definition |
|------|------------|
| **Validation** | Checking whether incoming data meets expected format, type, and constraints |
| **Transformation** | Converting or normalizing data into the format required by the server |
| **Syntactic Validation** | Checking whether data follows a specific structure or pattern |
| **Semantic Validation** | Checking whether data makes logical sense in the real-world context |
| **Type Validation** | Checking whether data is of the expected data type |
| **Pipeline** | A sequence of processing steps applied to data in order |
| **Query Parameter** | Key-value pairs in the URL after `?` (always strings by default) |
| **400 Bad Request** | HTTP status code indicating the client sent invalid data |
| **500 Internal Server Error** | HTTP status code indicating an unexpected server failure |
| **Controller Layer** | The layer that handles HTTP requests and responses |
| **Service Layer** | The layer that contains business logic |
| **Repository Layer** | The layer that interacts directly with the database |
