# HTTP Protocol Fundamentals

## Introduction to HTTP Protocol

Backend engineering is a massive field. If we tried to discuss every single component that could be part of it, we'd be stuck here for years. Instead, we'll focus on the topics used in the majority of codebases—let's say 90% of them.

With that in mind, let's talk about HTTP: the medium through which our browsers talk to our servers—either to send data or receive data from it. While there are many other ways and protocols that clients and servers use to communicate with each other, HTTP is one of the most widely used, so we'll focus on that.

Throughout our discussion, we can safely assume that HTTP and HTTPS are interchangeable. HTTPS, to oversimplify, is just a more secure version of HTTP—the underlying principles are the same, with additional security features like encryption, security certificates, and TLS. Those details border too much into network engineering, so we'll focus on the application layer.

## Two Core Ideas of HTTP

There are two ideas at the heart of the HTTP protocol.

### Statelessness

Statelessness means the server has no memory of past interactions. Each HTTP request carries all the necessary information for the server to process it—such as headers, URLs, and methods (which we'll see in a bit). After the server responds, it forgets about the request. If a client makes another request, the server treats it as a new and unrelated event.

This also means self-contained requests. Since the server doesn't remember past requests, each request must include all the necessary data—such as authentication tokens or session information—to handle that specific interaction. For example, when accessing a user profile, the client has to provide credentials like cookies or tokens on every request for the server to know which user is requesting the data.

#### Benefits of Statelessness

- **Simplicity**: Stateless design simplifies server architecture because the server doesn't need to store session information, which would otherwise require additional resources and complexity.
- **Scalability**: A stateless protocol makes it easy to distribute requests across multiple servers because no single server needs to keep track of a session.
- **Resilience**: If the server crashes, it doesn't affect the state of a client interaction since there's no session or memory of a request that needs to be restored.

That said, because HTTP is stateless, developers often implement state management techniques like cookies, sessions, or tokens to maintain continuity in interactions where needed—like user logins or shopping carts. We'll explore those soon in this series.

### Client-Server Model

In a typical HTTP request flow, there is always a client and a server.

- **Client**: Typically a web browser or an application that initiates communication by sending a request to the server. The client is responsible for providing all information needed by the server—such as the URL of the resource, headers, and everything else.
- **Server**: Hosts resources like websites, APIs, or other content and waits for incoming requests from clients. When the server receives a request, it processes it and sends back the appropriate response—such as a web page, data, an error message, a JSON file, a text file, or any other kind of content.

The key thing to remember is that the HTTP protocol states that communication is always initiated by the client to get some kind of response from the server.

## HTTP and Transport Layer

To send some kind of request or receive some kind of response, first the client and the server need to establish some kind of connection mechanism. Otherwise, how are they going to communicate? What is the medium of communication?

For this, HTTP uses TCP (Transmission Control Protocol). HTTP doesn't require the underlying transport protocol to be connection-based—it only requires it to be reliable and not lose messages (at minimum presenting an error in such cases). Among the two most common transport protocols on the internet—TCP and UDP—TCP is considered more reliable. HTTP therefore relies on the TCP standard, which is connection-based.

This is called the OSI model, often referenced when talking about sending and receiving data over a network. As backend engineers, we often deal with Layer 7—the application layer. A lot of discussions—like the TCP handshake, establishing connections, and TLS encryptions—often border into these layers, so they're mostly network engineering concepts. While it's good to know about them, if we start exploring them, it'll be a rabbit hole and we'd have to cover a lot of concepts.

What we'll do is focus on the application layer with a brief overview of what happens in the network layer. As I mentioned, HTTP uses TCP, and TCP uses something like a 3-way handshake. If you're curious about that, you can look it up and study more on your own.

The key takeaway from all these discussions is that clients and servers establish some kind of network connection, and messages are sent and received. That's all you need to remember for now.

## HTTP Versions Evolution

Throughout the years, we've had different versions of HTTP that kept redefining how clients and servers send and receive data.

### HTTP 1.0

In HTTP 1.0, each request opened a new connection. This led to inefficiencies since a connection had to be established and closed for every request and response, which slowed performance.

### HTTP 1.1

HTTP 1.1 introduced **persistent connections**, allowing multiple requests and responses over the same TCP connection that was established before sending the request. This significantly improved performance. It also added chunk transfer encoding and better caching mechanisms.

### HTTP 2.0

HTTP 2.0 introduced **multiplexing**, allowing multiple requests or responses over a single connection. It uses binary framing instead of text and supports header compression with HPACK, as well as server push—allowing servers to send resources before the client requests them.

### HTTP 3.0

HTTP 3.0 is built on the QUIC protocol, a transport layer protocol over UDP instead of TCP. It's designed to improve performance with faster connection establishment, reduced latency, and better handling of packet loss. It also continues to support multiplexing without head-of-line blocking, which is still an issue in HTTP 2.0.

Again, if you get too much into the network stuff, it's a rabbit hole. All we need to remember is that clients and servers establish some kind of network connection, and messages are sent and received.

## HTTP Messages

Now that we've mentioned messages, let's look at what HTTP messages look like. There are two types: request messages (sent by the client) and response messages (received by the client from the server).

### Request Message Structure

A request message includes:

- **Request method**: The type of action (GET, POST, etc.)
- **Resource URL**: The endpoint being requested from the server
- **HTTP version**: E.g., HTTP/1.1 (currently the most used)
- **Host**: The domain of the frontend
- **Headers**: Key-value pairs of parameters
- **Blank line**: Signifies the end of headers
- **Request body**: Some information the client wants to send to the server

### Response Message Structure

A response message includes:

- **HTTP version**
- **Status code**: E.g., 200 OK
- **Response headers**: Key-value pairs of parameters
- **Blank line**: Signifies the end of headers
- **Response body**: The actual content

## HTTP Headers

HTTP headers are a critical component of both requests and responses. Let's dive into them.

### What Are Headers and Why Do We Need Them?

On a high level, headers are basically key-value pairs of different parameters that are sent in a request or received in a response.

The next question is: why do we need headers? Why not send all these values in the URL or request body? Why create a different section just for sending metadata about the request or response?

To understand this, let's take a real-life example. When we send parcels, we have the address, the recipient's phone number, the state, the PIN code, and other details. Do we keep this information inside the package, or do we write it on top? We write it on top. Why? Because the people taking the parcel from the sender to the receiver need to know different information in order to successfully transmit the package through different modes of transmission. If we kept all that information inside the parcel, they'd have to open it every time—which doesn't make sense.

Keeping the metadata—the address and recipient information—on top of the parcel gives a quick way to check different metadata about the package. HTTP headers can be thought of in the same way, but they have more uses.

### Categories of HTTP Headers

There are many different types of headers, so let's categorize them.

#### 1. Request Headers

Request headers are sent by the client to the server to provide information about the request itself.

- **User-Agent**: Identifies what kind of client is making the request—a browser, Postman, a server, or a mobile app.
- **Authorization**: Sends different credentials like bearer tokens to the server to identify the user.
- **Accept**: Provides information about what kind of content the client is expecting—whether it's JSON, text, or an HTML file.

Request headers help the server understand the client's environment, its preferences, and its capabilities.

#### 2. General Headers

General headers are used in both requests and responses and have metadata about the message itself.

- **Date**: The timestamp of the message.
- **Caching mechanisms**: Like no-cache or max-age.
- **Connection**: Whether to keep it alive or close it.

General headers provide information about the request message or the response message itself.

#### 3. Representation Headers

Representation headers primarily deal with the representation of the resource being transmitted—whether it's a request body or a response message.

- **Content-Type**: Describes the media type of the request or response—it could be JSON or HTML.
- **Content-Length**: Describes the size of the resource in bytes.
- **Content-Encoding**: Specifies any encoding, like gzip or deflate.
- **ETag**: A unique identifier, mostly used for caching.

Representation headers provide information about the body of the message, ensuring that clients and servers know how to interpret and process the request or response.

#### 4. Security Headers

Security headers are used to enhance the security of the request and response by controlling behaviors like content loading, cookies, and encryption.

- **HSTS (HTTP Strict Transport Security)**: Ensures that the client only communicates with the server over HTTPS, preventing protocol downgrade attacks.
- **CSP (Content Security Policy)**: Restricts the sources from which content like JavaScript, CSS, and images can be loaded, helping prevent cross-site scripting attacks.
- **X-Frame-Options**: Prevents the webpage from being embedded in an iframe, mitigating clickjacking attacks.
- **X-Content-Type-Options**: Ensures that the browser does not try to guess the MIME type of the content, preventing MIME type sniffing attacks.
- **Cookie flags**: HttpOnly (cookies inaccessible with JavaScript) and Secure (cookies sent only over HTTPS).

Security headers help protect the client and server from a variety of attacks by controlling how the browser behaves with resources and enforcing security policies.

### Two Key Ideas in HTTP Headers

#### 1. Extensibility

HTTP is highly extensible because headers can be easily added or customized without altering the underlying protocol. We only have to add some kind of metadata, and the whole flow of the interaction changes depending on that.

Headers can be defined and used for various purposes, making HTTP adaptable to new technologies and use cases. For instance:
- Security enhancements: Headers like Strict-Transport-Security force secure connections.
- Custom headers: Developers can create custom headers like `X-custom-header` for specific application use cases.
- Content negotiation: The Accept, Accept-Language, and Accept-Encoding headers allow servers to serve different versions of content depending on client preferences.

#### 2. Remote Control

HTTP headers act as a kind of remote control on the server side. They allow the client to send instructions or preferences to the server, influencing how the server responds or processes requests.

- **Content type negotiation**: Clients can request specific formats using the Accept header. If the client says "I want HTML format," the server sends HTML. If the client says "I want JSON," the server sends JSON.
- **Caching and expiration control**: The server can use headers like Cache-Control or Expires to control how long a resource should be cached by the client.
- **Authentication**: The client can authenticate itself to the server through the Authorization header, influencing access control decisions.

These provide more capabilities on top of our messages, which are very useful in many instances.

## HTTP Methods

HTTP methods exist to represent different kinds of actions that a client—like a browser or an API consumer—can request on a server. Instead of every request doing the same thing, methods define the intent of the interaction. The keyword here is "intent"—this gives clear semantic meaning to each type of action, and it's pretty intuitive.

### Common HTTP Methods

- **GET**: Used to fetch some kind of data from the server. It should not modify anything on the server.
- **POST**: Used to create some data in the server. POST requests have a body—which makes sense, because how else would you send user data to the server?
- **PATCH**: Used to update some data. For example, on a user's profile page where they can update their name. This also has a request body to send the data to the server.
- **PUT**: Also used to update data, but what sets it apart from PATCH is that whatever data comes in the request body should completely replace the previous instance. Basically, PATCH can be thought of as an append action or selective replacement, while PUT is a complete replacement.
  - **Thumb rule**: Always use PATCH unless you have a specific use case for PUT. A lot of times developers use PUT when they should be using PATCH, going against the semantics.
- **DELETE**: As the name suggests, used to delete some kind of resource from the server.

### Idempotent vs Non-Idempotent Methods

There's a prevalent idea in the context of HTTP methods: idempotence. Idempotent methods can be called multiple times, and we can expect the same kind of result.

- **GET**: Obviously idempotent. You're trying to fetch data from the server—it doesn't matter how many times you fetch it, the data should be the same. You should not be able to modify any data on the server.
- **PUT**: Idempotent because PUT completely replaces the resource in the server. It doesn't matter how many times you replace the old with the new data—the result will be the same.
- **DELETE**: Idempotent. You can only delete a resource from the server once. After that, it's already deleted. You cannot perform the action multiple times and expect different results—the resource can only be deleted once.
- **POST**: Generally considered non-idempotent. Once you submit a request to create data—let's say a user creates a note using your app—the first time they submit the request, it goes through successfully and a new note is created. When they do it a second time, we create a new note. There are two different results for the same kind of request, which is why we consider POST to be a non-idempotent method.

### OPTIONS Method

After all the HTTP methods, there's one other method called OPTIONS. This has a very interesting use case—it's used in the CORS flow, which is part of the same-origin policy that browsers have.

You probably won't use the OPTIONS method directly as a developer, but you'll see them once in a while in your browser network tab in pre-flight requests. The OPTIONS method is used to fetch the capabilities of the server for a cross-origin request—also known as CORS.

## CORS (Cross-Origin Resource Sharing)

Browsers have something called the same-origin policy. By default, browsers follow the same-origin policy, which restricts web pages from making requests to a domain different from the one serving the web page.

CORS is a security mechanism enforced by browsers to control how web applications interact with resources hosted on different domains (cross-origin). Without CORS, browsers block requests made from a web application running on one origin (like example.com) to a different origin (like api.another-example.com) for security reasons.

CORS allows servers to specify who can access their resources and how.

### CORS Flow Types

There are primarily two types of flows in CORS: simple request flow and pre-flight request flow.

#### Simple Request Flow

Imagine our client (frontend) is at the domain `example.com` and our server is at the domain `api.example.com`. We make a GET request.

**The flow:**

1. The client sends a request. The browser automatically adds the Origin header to indicate the origin of the request.
2. The request uses a simple method (usually GET, POST, or HEAD).
3. The request reaches the server.
4. The server checks the Origin header against its CORS policy.
5. If the origin is allowed, the server includes the Access-Control-Allow-Origin header in the response.
6. The server responds with the resource and includes the necessary CORS headers—for example, Access-Control-Allow-Origin.

**Browser validation:**

When the browser receives the response, it sees that our domain (`example.com`) is different from the host domain (`api.another-example.com`). Since these two are different, the browser checks whether the server's response has the Access-Control-Allow-Origin header. If the server responds with the domain of the client (`example.com`) or with `*` (which means allow all origins), then the browser lets the response through to the JavaScript client requesting the resource.

If the server does not add the corresponding CORS headers or does not allow this particular domain as a client, it excludes this header from the response. When the browser parses the response and sees the absence of this header, it blocks the response from being passed. You'll get an error in your console, and you'll see the error in the network tab as a CORS error.

#### Pre-Flight Request Flow

How does a browser distinguish between a simple request flow and a pre-flight request flow? The browser checks three conditions before deciding it has to do a pre-flight request (which means it has to do a request before the original request to inquire about capabilities and let the server know about some capabilities).

A request qualifies as a pre-flight request if:
1. It's a cross-origin request (our domain and server domain are different)
2. **AND** one of these three conditions is true:
   - The method is not GET, POST, or HEAD (e.g., it's a PUT request or DELETE request)
   - The request includes some non-simple headers (anything apart from general headers or request headers—for example, an authorization header or custom headers)
   - The request has a content type other than application/x-www-form-urlencoded, multipart/form-data, or text/plain (these are called general content types or simple content types)

If our application client requests data in JSON, it will be considered a pre-flight request. If you're a frontend or backend engineer, you know that mostly we deal with JSON data, so most of our requests are considered pre-flight requests.

**What does a pre-flight request look like?**

This is when we finally see the use of the OPTIONS method. A pre-flight request is made with the OPTIONS method:

- Method: OPTIONS
- Resource URL
- HTTP version
- Host header (the header of our API)
- Origin (the domain of the frontend)
- Access-Control-Request-Method: Asking the server whether this method is supported for this URL (e.g., "I'm making a cross-origin request, do you support this PUT method for this route?")
- Access-Control-Request-Headers: Asking whether the server supports a particular header if needed

The browser sends the OPTIONS request to the server. This request does not include actual data—no request body or anything. It's just a general inquiry to the server about its capabilities.

**Server response to pre-flight:**

If the server properly handles CORS requests and supports cross-origin requests, it responds with:

- Status code 204 (no content)—just general information
- Access-Control-Allow-Origin: "I allow the client's domain as a valid cross-origin request" (can respond with the specific domain or `*` for all origins)
- Access-Control-Allow-Methods: Lists allowed methods for this resource (e.g., GET, POST, PUT, DELETE)
- Access-Control-Allow-Headers: Lists allowed non-simple headers (e.g., content-type, authorization)
- Access-Control-Max-Age: "Don't make any more pre-flight requests to me for at least the next 24 hours—these configs will be the same" (saves bandwidth for both servers and clients)

The browser checks off all these conditions, then sends the final request—the original request that the client wanted to make. The server responds according to the original request with whatever operations are required.

This is how the typical CORS flow works.

## HTTP Status Codes

HTTP response codes exist to communicate the result of a request in a standardized way. You can just look at the response code and see whether the request was successful or not, or what the state of the server is—without looking into the body or judging from the response message.

They quickly inform the client whether the request was successful, resulted in an error, or requires further action. They also help clients handle errors by providing specific codes to identify the problem. For example, unauthorized access will return 401, so clients can check for 401 and log the user out, saying they have to log in again. Or if it's a bad request error due to invalid form submission, the client can ask the user to make changes and resubmit.

HTTP response codes are standardized across all web services, enabling consistency in how servers communicate with different clients regardless of the platform or language used. It doesn't matter whether you're making a server in Python, Go, Rust, JavaScript, or Ruby—you have to follow this standard. If the request was successful, you have to return 200. If you created something, you have to return 201.

Before HTTP status codes, clients would have to guess the outcome of a request based on the content of the response, leading to inconsistencies and inefficiencies. HTTP status codes solve this by providing a universal language that all clients and servers understand, streamlining interactions and error handling.

### Status Code Categories

On a high level, response codes are three-digit numbers. Depending on the starting digit, we categorize them:

- **1xx**: Informational responses
- **2xx**: Success responses
- **3xx**: Redirection
- **4xx**: Client errors
- **5xx**: Server errors

### Common Status Codes

#### 1xx Informational

These codes are sent by the server to indicate that it has received the headers and the client can proceed to send the request body. They're commonly used in large uploads—the client sends the headers first, and if the server is okay with the request, it sends a 100 Continue so the client can send the rest of the body.

- **100 Continue**: Server received headers, client can proceed to send body
- **101 Switching Protocols**: Server is switching protocols as requested by the client (e.g., upgrading from HTTP to WebSocket)

These aren't the most commonly used response codes you'll encounter in day-to-day work.

#### 2xx Success

- **200 OK**: The most common code. Indicates that the request was successful and the server is returning the requested resource or performing the requested action. Example: successful GET request where a resource is retrieved.
- **201 Created**: Indicates that the request has been fulfilled and resulted in the creation of a new resource. Example: POST request or new form submissions—that's where servers use this response to indicate that a new resource has been created.
- **204 No Content**: Indicates that the request was successful but there is no content to return. We see this in CORS flow when we send an OPTIONS request (pre-flight)—the server responds with 204 saying there's no content, but the information is in the form of headers. We also sometimes use it for DELETE requests—the client makes a delete request, the server deletes the resource, and responds with 204 saying "okay, I've deleted it but there's no content to return."

#### 3xx Redirection

- **301 Moved Permanently**: The requested resource has been permanently moved to a new URL, and future requests should use this new URL. Example: Initially you had a route called `/user`, and eventually you decided to move that route to `/person`. To maintain backwards compatibility so that old users or applications still using the old route don't break, you add a 301 response for that route and redirect to the `/person` route. It's a permanent redirect.
- **302 Found (Temporary Redirect)**: The requested resource is temporarily located at a different URL, but the client should continue to use the original URL for future requests. Example: You're running a campaign, and for a couple of hours you want to redirect a particular route to a new route to catch new traffic or show a different UI, but you don't want to stick to that. You want to revert the changes, so you tell the client "for now I'm making a redirect, but later on you should use the original route only."
- **304 Not Modified**: Indicates that the resource has not been modified since the last time the client requested it. This is mostly used in conjunction with conditional GET requests to allow efficient caching—when using ETags to let the client know that the response is not modified, they should use the cached version instead of downloading the new response.

#### 4xx Client Errors

As a backend engineer, you'll mostly deal with these errors because these are client errors—errors triggered by some behavior from the client.

- **400 Bad Request**: Triggered when the client sends invalid data or illogical data. Example: You're expecting a number and the client sends an array or a string, or you're expecting an email and the client sends a phone number. You're letting the client know that there's something wrong with the request format—fix it and make a new request.
- **401 Unauthorized**: Triggered when a request requires authentication but the client has either failed to provide valid credentials or is not authenticated at all. Example: You're expecting a JWT token, and either the JWT token has expired or the client hasn't sent the token in the first place. In those scenarios, you respond with 401 to say "you're unauthorized."
- **403 Forbidden**: The server understood the request but refuses to authorize. This can happen even if the client is authenticated. Example: When a user tries to access a resource they don't have permission to access—user A trying to delete a resource of user B. The server says "you don't have the necessary permissions to perform this action—you're forbidden."
- **404 Not Found**: The most famous status code. Fired when the client requests a resource that's unavailable—either because the URL is incorrect or the resource has been deleted.
- **405 Method Not Allowed**: Fired when an invalid HTTP method is used, such as trying to PUT to a resource that only accepts GET or POST. This often happens because of typos—you're working on the frontend and instead of doing a PUT request you do a PATCH request, or instead of POST you do PUT. The server says "method not allowed."
- **409 Conflict**: Has a number of use cases. Example: In your app, you're allowing users to create folders, and the condition is that folder names must be be unique—they cannot create two folders with the same name. When they try to create a new folder and submit a POST request, you check whether the folder already exists. If it does, you respond with 409 conflict so the client understands that a folder with that name already exists and should try with a new folder name.
- **429 Too Many Requests**: Used when rate limiting clients. If a client tries to make too many requests in a particular interval—let's say you've configured your server to allow at most 60 requests for a client in 1 second, and the client tries to exceed that—you respond with 429 too many requests.

#### 5xx Server Errors

- **500 Internal Server Error**: The most famous one. Used for unexpected conditions in the server—some process broke or some exceptions were raised that weren't handled in the server. Something unexpected happened at the server. Instead of just returning an empty response or breaking or hanging the request, we respond with 500 internal server error so the client knows something went wrong with the server.
- **501 Not Implemented**: Used when the server does not support the requested HTTP method or functionality but plans to add it soon. You respond with 501 to indicate "currently it is not supported, it might be in the future"—you're trying to get that intention through to the client.
- **502 Bad Gateway**: We usually see this in proxies like Nginx. It means when a server acts as a proxy (like in a load balancing system or reverse proxy), the upstream server returns an invalid response. This is not something we return intentionally—it's mostly handled by proxies and load balancers.
- **503 Service Unavailable**: When the service is down—temporarily unable to handle the request, such as during high traffic or when going into maintenance. You return 503 to let the client know that the service is unavailable right now and they should try again later.
- **504 Gateway Timeout**: Similar to 502, but this specifically means that the upstream server failed to respond within the timeout period. Example: You're using Nginx, and Nginx could not get a response from your original server running behind it. Nginx responds with 504 gateway timeout—it did not receive any response from your original server, hence "timeout."

That's pretty much all the response codes you need to know to work with 95% of use cases.

## HTTP Caching

HTTP caching is a technique to store copies of responses for reuse, reducing the need for repeated requests to the server. This improves load time, reduces bandwidth, and decreases server load—because the client doesn't need to download a lot of data, and the server doesn't need to send a lot of data if the data hasn't changed. The client can just reuse the old data—that's what caching means: reusing the old data if the data hasn't changed.

### Caching Headers

There are three important headers used for caching:

- **Cache-Control**: Tells the client how long to maintain the cache for this resource (e.g., max-age=10 seconds)
- **ETag**: Basically a hash—usually computed from a response (the server might take the response, hash it, and send that hash in the form of an ETag)
- **Last-Modified**: The last time this resource was modified—judging from this time and date, the client can decide whether to use the old cached resource or request a new one

### Caching Flow

1. **Initial request**: Client makes a GET request to the endpoint. Server responds with 200, the requested resource, and the caching headers (Cache-Control, ETag, Last-Modified).
2. **Subsequent request**: Client makes another GET request to the same endpoint. Now the headers to focus on in the request are:
   - **If-None-Match**: The ETag the client has cached
   - **If-Modified-Since**: The last modified timestamp the client has cached
   
   The client is saying to the server: "If the ETag (hashed version) of the requested resource is not the same as this one I have, or if the resource has been modified after this date (meaning I have the outdated version), send me a new resource. Otherwise, I'll just use my cached version."
3. **Server response**: The server checks the If-None-Match header (the ETag) and also checks the Last-Modified. Since the resource matched either the ETag or the Last-Modified, the server sends the client a 304 response—Not Modified. This means the requested resource has not been modified since the last time you fetched it, so you can go ahead and use your cached version.
4. **After update**: If the resource is updated (e.g., via a POST/PUT request), the server responds with 200 and a new ETag. The client then uses this new ETag for subsequent requests. If the client sends the old ETag, the server responds with 200 instead of 304 because the resource has been modified, and sends the updated ETag and Last-Modified.

This is how you handle caching using the HTTP protocol with different headers in server and client.

**Note**: In a production setting, HTTP-based caching gets complicated because the server has to manually implement and manage all these ETags. If you forget to update an ETag by mistake, the client will continue to use the cached version with the outdated resource, which is not a good idea. Nowadays we have better solutions for caching—for example, React Query, which is a complete client-side caching solution. The client has complete power over when it wants to use a cached resource and when it wants to refetch, at what interval, and other powerful capabilities. In my opinion, that's a much better solution compared to traditional HTTP-based caching. But it's good to know that we have this HTTP-based caching option if our use cases are simple enough.

## Content Negotiation

Content negotiation is an important topic that comes up in the client-server model in HTTP. It's a mechanism using which clients and servers exchange information about different types of content, encoding, and representation of the content.

Basically, it's a mechanism for the client and server to agree on the best format to exchange data. The client can indicate its preferred format—like JSON or XML or HTML—and the server will try to respond with a compatible format, or if not available, a fallback format.

### Types of Content Negotiation

1. **Media type negotiation**: The client specifies the desired format through the Accept header (e.g., application/json or XML).
2. **Language negotiation**: The client requests content in a specific language using the Accept-Language header (e.g., English or Spanish).
3. **Encoding negotiation**: The client specifies which encoding it supports using the Accept-Encoding header (e.g., gzip or deflate).

### HTTP Compression

HTTP compression falls under the content negotiation umbrella. Let's look at why we need compression and how it works.

**Example**: Replace the JSON response with a very large file of 11,000 entries. When you fire the request with compression enabled on the server side (gzip encoding), the response size is 3.8 MB. If you disable compression and fire the same request, the size becomes 26 MB—that's a huge increase. Imagine every client having to download that file—it causes a lot of bandwidth usage.

That's why we need compression: if the response size is very large, we can compress it to a format, and on the client side the browser can decompress it and get the same response.

## Persistent Connections and Keep-Alive

In the early days of HTTP (specifically HTTP 1.0), each request-response cycle required a separate connection to the server. This created inefficiencies since establishing and closing TCP connections is resource-intensive and slow.

To address this, persistent connections were introduced in HTTP 1.1. With persistent connections, a single TCP connection can be reused for multiple requests and responses, avoiding the overhead of opening and closing a connection for every interaction.

### Keep-Alive Header

Keep-Alive is the mechanism that enables persistent connections. It allows the client and server to reuse the same connection for multiple request-responses until one of them decides to close it.

**Key points to remember:**

1. In HTTP 1.1, connections are persistent by default—you don't have to do anything explicitly about them. They remain open for further requests unless explicitly closed.
2. Multiple requests and responses can be sent over a single connection. This reduces latency and saves resources as fewer connections need to be established.
3. The Connection: keep-alive header is still sometimes used to explicitly ask the server to keep the connection open. This header can also include options like how long the connection should remain open (with a timeout) or how many requests can be sent before the connection is closed (with a max value).
4. When Connection is set to close, the connection is closed after the response is sent. This is the behavior in HTTP 1.0 by default, and it can still be explicitly enforced in HTTP 1.1.

This is some information you just have to understand, but you usually won't be working with them explicitly—the default values work fine.

## Handling Large Requests and Responses

Let's look at how servers take in large requests (files, video files, image files, audio files—anything very large compared to our typical JSON) and how clients can receive large responses.

### Large Requests: Multipart/form-data

Multipart is usually used for sending large files or any kind of files to the server from the client. The difference from our typical JSON request body is that in a multipart request, the file data—the binary data—is transferred to the server in parts, in different parts—that's why it's called "multipart."

**Request structure:**
- POST method
- Content-Length header
- Content-Type: multipart/form-data
- **Boundary parameter**: Since the binary data is transferred in parts, we need to specify what will be the delimiter—what will separate the parts. We specify a code that will separate the parts. The delimiter appears at the start of the binary data and again at the end when all the binary data ends in the request body.

The server reads the file and responds with details of the file to indicate that the upload was successful. The idea is: whenever you want to transfer large files to the server, you should use multipart requests.

### Large Responses: Streaming

For receiving large responses from the server, we stream the data to the client side in chunks.

**Response structure:**
- Content-Type: text/event-stream (indicates it will stream the data to the client through different events)
- Connection: keep-alive (keep the connection alive until all the data is sent)

The client keeps receiving chunks continuously from the server in different requests. The client keeps appending all the data it's receiving from the server and constructing the whole file. This is how we transfer a large file from a server to a client using chunk transfer or text/event-stream.

## SSL/TLS/HTTPS Overview

Before we end, let's give a brief idea about what SSL, TLS, and HTTPS mean. Even though we don't explicitly work with them, it's good to know what they are.

### SSL (Secure Sockets Layer)

SSL was the original protocol for securing communications between a client (like a web browser) and a server. It encrypts data so that sensitive information like passwords or credit card numbers cannot be intercepted by attackers. It was the original encryption mechanism between clients and servers.

Currently, SSL is outdated due to some security vulnerabilities and has been replaced by TLS.

### TLS (Transport Layer Security)

TLS is the modern and more secure version of SSL. It encrypts data in transit, ensuring that any data sent between the client and server is protected from interception and tampering.

TLS uses certificates to authenticate the server and establish an encrypted connection, preventing eavesdropping and data breaches. TLS is continuously updated with newer versions offering better security. The current recommended version is TLS 1.3.

### HTTPS

HTTPS is basically HTTP with more security features—provided by SSL (originally) or TLS (currently). The underlying mechanism is TLS, and HTTPS is the one that uses TLS.

How it works: When you visit a website using HTTPS, TLS encrypts the communication between your browser and the server. This protects sensitive data like login credentials from being intercepted by attackers.
