# Authentication & Authorization: A Complete Guide

> **Prerequisites:** Client-server architecture, HTTP, REST APIs.

---

## 1. The Two Core Questions

| Question | Concept | Meaning |
|----------|---------|---------|
| Who are you? | **Authentication** | Verifying identity |
| What can you do? | **Authorization** | Checking permissions |

**Example:** Logging into a note-taking app.
- **Authentication:** Email/password correct? "Yes, you are Alice."
- **Authorization:** Alice can create/delete her own notes, but cannot access admin dashboards.

---

## 2. A Brief History of Authentication

### Pre-Industrial: Trust-Based
- Village elders vouched for people; handshakes sealed deals.
- **Problem:** Does not scale beyond small communities.

### Medieval: Physical Tokens (Seals)
- Wax seals with unique patterns on documents.
- **Principle:** *Something you have*.
- **Problem:** Forgery — the first **authentication bypass attacks**.

### Industrial Revolution: Passphrases
- Telegraph operators used pre-agreed static passphrases (early passwords).
- **Principle:** *Something you know*.

### 1961: Digital Passwords (MIT CTSS)
- First multi-user password system.
- **Vulnerability:** Stored in **plain text**.
- **Incident:** Someone printed the password file, exposing all credentials.
- **Lesson:** Birth of **secure password storage** and **hashing**.

### 1970s: Asymmetric Cryptography
- Diffie-Hellman key exchange enabled shared secrets over untrusted networks.
- **Kerberos:** Ticket-based authentication via trusted third parties.

### 1990s: Multi-Factor Authentication (MFA)
- Combine proof factors:
  - **Something you know** (password)
  - **Something you have** (OTP generator)
  - **Something you are** (biometric)

### 21st Century
- **OAuth 2.0** — delegated authorization
- **JWT** — stateless tokens
- **Zero Trust** — never trust, always verify
- **Passwordless** — WebAuthn with hardware keys

### The Future
- **Decentralized Identity** (blockchain)
- **Post-Quantum Cryptography** — resistant to quantum computers

## 3. Three Core Components

### 3.1 Sessions

**The Problem:** HTTP is **stateless**. Every request is isolated with no memory.

**The Solution:** Sessions establish temporary server-side context for each user.

**How Sessions Work:**

```
1. User logs in (email/password) -> Server
2. Server creates unique Session ID, stores user data in Redis/DB
3. Server sends Session ID back to client as a Cookie
4. Every subsequent request automatically includes the Cookie
5. Server looks up Session ID in Redis/DB to identify the user
```

**Session Storage Evolution:**

| Era | Storage | Characteristics |
|-----|---------|---------------|
| Early | File-based | Simple, not scalable |
| Growth | Database | Persistent across restarts |
| Modern | Redis/Memcached | Fast in-memory lookups |

**Key Properties:**
- Sessions are **short-lived** (expire after a set time).
- The server can **revoke instantly** by deleting the session.

---

### 3.2 JWT (JSON Web Tokens)

**Why JWTs Emerged:** Maintaining millions of sessions became costly. Synchronizing sessions across distributed servers introduced latency.

**What is a JWT?**
A **stateless**, self-contained token carrying user claims and a cryptographic signature. It is Base64-encoded with three parts:

```
xxxxx.yyyyy.zzzzz
(header).(payload).(signature)
```

**JWT Parts:**

| Part | Content | Purpose |
|------|---------|---------|
| **Header** | Signing algorithm metadata | Tells the server how to verify |
| **Payload** | User claims: `sub` (user ID), `iat` (issued at), `name`, `role` | Carries the data |
| **Signature** | Cryptographic hash | Verifies the token was not tampered with |

**Advantages of JWT:**
1. **Statelessness** — no server-side storage needed
2. **Scalability** — any server with the secret key can verify the token
3. **Portability** — compact, URL-friendly, works across systems

**Disadvantages of JWT:**
1. **No revocation** — cannot invalidate a token before it expires (unless you change the secret, which logs out ALL users)
2. **Theft risk** — if stolen, attacker can impersonate the user until expiry

**Hybrid Approach:**
Many systems verify the JWT first, then check a **blacklist** (Redis/DB) for revoked tokens. This adds a lookup step, raising the question: if you are doing lookups anyway, why not use stateful sessions? The answer depends on your architecture needs.

> **Advice:** For production systems, consider using an **Auth Provider** (Auth0, Clerk) rather than building custom auth.

---

### 3.3 Cookies

A **cookie** is a piece of data stored in the user's browser by the server.

**Key Properties:**
- Set by the server via HTTP response headers
- Automatically sent back with every subsequent request to that domain
- Scoped to the originating server (one server cannot read another's cookies)
- `HttpOnly` cookies cannot be accessed by JavaScript (security feature)

**In Authentication Flow:**
```
1. Client sends login credentials
2. Server validates, creates session/JWT
3. Server sets cookie with Session ID or JWT token
4. Browser auto-attaches cookie to all future requests
5. Server reads cookie, validates token/session
```

## 4. Types of Authentication

### 4.1 Stateful Authentication (Sessions)

**Flow:**
```
1. Client sends username/password
2. Server validates, generates Session ID
3. Server stores {Session ID: user_data} in Redis
4. Server sends Session ID in HttpOnly Cookie
5. Client sends cookie with every request
6. Server checks Redis for Session ID -> identifies user
```

**Pros:**
- Centralized control over all active sessions
- Real-time session information
- Easy revocation — delete the session from Redis

**Cons:**
- Scalability challenges in distributed systems
- Higher operational complexity across regions

**Best for:** Web apps with moderate traffic and strict session control.

---

### 4.2 Stateless Authentication (JWT)

**Flow:**
```
1. Client sends username/password
2. Server validates, generates signed JWT with secret key
3. Server sends JWT back to client
4. Client stores JWT (localStorage/cookie) and sends in Authorization header
5. Server verifies JWT signature with secret key -> identifies user
```

**Pros:**
- Highly scalable — no session store needed
- Ideal for distributed architectures and microservices
- Mobile-friendly

**Cons:**
- Token revocation is complex
- If token is stolen, it is valid until expiry

**Best for:** APIs, distributed systems, mobile apps.

---

### 4.3 API Key Authentication

**What it is:** A cryptographically random string generated by the platform that grants programmatic access to the API.

**Use Case:** Machine-to-machine (M2M) communication — one server talking to another server programmatically (no human UI involved).

**Example:**
- ChatGPT has a web UI for humans, but also provides **API keys**.
- Your backend server can use the API key to call OpenAI's servers directly.
- The API key identifies your account, tracks quota/usage, and enforces permissions.

**When to use:**
- Server-to-server communication
- Third-party integrations
- Single-purpose client access to APIs

**Why not use sessions/JWT for M2M?**
Session/JWT flows require human triggers (login forms, credential entry). API keys are static secrets stored in environment variables, ideal for automated programmatic access.

---

### 4.4 OAuth 2.0 & OpenID Connect

**The Problem (Delegation):**
One platform needs access to resources on another platform.
- A travel app needs to scan Gmail for flight tickets.
- A social app wants to import Google contacts.

**The Old (Bad) Solution:** Users shared passwords — giving full access to everything.

**The New Solution (OAuth 2.0):**
Share **tokens with limited permissions** instead of passwords.

**OAuth 2.0 Roles:**

| Role | Example | Description |
|------|---------|-------------|
| **Resource Owner** | You | The user who owns the data |
| **Client** | Travel App | The app requesting access |
| **Resource Server** | Gmail/Google | The server holding the data |
| **Authorization Server** | Google Auth | The server that issues tokens |

**OAuth 2.0 Flow (Simplified):**
```
1. Client redirects user to Authorization Server
2. User logs in and grants specific permissions (e.g., "read contacts only")
3. Authorization Server sends token to Client
4. Client uses token to access Resource Server (e.g., fetch contacts)
```

**OAuth 2.0 Grant Types:**

| Flow | Use Case |
|------|----------|
| **Authorization Code** | Server-side web apps |
| **Implicit** | Browser apps (now discouraged) |
| **Client Credentials** | Machine-to-machine |
| **Device Code** | Smart TVs, limited-input devices |

**OpenID Connect (OIDC):**
Built on top of OAuth 2.0 to add **authentication** (identity).
- OAuth 2.0 solves **authorization** (what you can access).
- OIDC adds an **ID Token** (a JWT) containing user identity: name, email, user ID.
- This powers "Sign in with Google / Facebook / Discord".

**OIDC Flow:**
```
1. User clicks "Sign in with Google"
2. Redirected to Google's Authorization Server
3. User logs in, grants permissions
4. Authorization Server sends Authorization Code + ID Token
5. Client exchanges code for Access Token
6. Client uses Access Token to access resources (e.g., Google Keep notes)
7. ID Token contains user's identity info (name, email)
```

## 5. When to Use Which Authentication Type

| Type | Best For |
|------|----------|
| **Stateful (Sessions)** | Web apps with moderate traffic, strict session control, easy revocation needs |
| **Stateless (JWT)** | APIs, distributed systems, microservices, mobile apps |
| **API Keys** | Machine-to-machine communication, server-to-server integrations, third-party API access |
| **OAuth 2.0 + OIDC** | Third-party logins (Sign in with Google), delegated resource access, SSO |

**Hybrid Approach:**
Many platforms use both:
- **Stateful sessions** for their main web app (browser clients)
- **Stateless JWTs** for mobile apps and third-party API integrations

## 6. Authorization: Role-Based Access Control (RBAC)

**The Problem:**
Not all users should have the same permissions. A regular user should not access admin-only features (e.g., viewing deleted notes in a "Dead Zone").

**What is Authorization?**
The process of determining **what a user can do** after they have been authenticated.

**RBAC (Role-Based Access Control):**
The most common authorization model. Users are assigned **roles**, and roles are assigned **permissions**.

**Typical Roles:**
- **User** — read/write own content
- **Moderator** — read/write + moderate content
- **Admin** — read/write + access to admin panels + view deleted content

**How it works in practice:**
```
1. User logs in (authentication)
2. Server determines user's role (User, Admin, etc.)
3. Server attaches role info to the request context (middleware)
4. When user requests an admin-only API:
   - If role is Admin -> allow access
   - If role is User -> return 403 Forbidden
```

**Granularity:**
You can go as granular as needed:
- Role-level: "Admins can access Dead Zone"
- Resource-level: "User can edit Note #123 but not Note #456"
- Action-level: "User can read but cannot delete"

## 7. Security Considerations for Backend Engineers

### 7.1 Generic Error Messages

**The Mistake:**
```
User not found          -> Attacker learns: username does not exist
Incorrect password      -> Attacker learns: username is correct
Account locked          -> Attacker learns: username exists AND password was correct
```

**The Rule:**
Always return **generic** authentication error messages:
```
Authentication failed
Invalid credentials
```

Do not reveal whether the username or password was the failing factor. This prevents attackers from narrowing down their attack strategy.

---

### 7.2 Timing Attacks

**The Problem:**
In a typical login flow:
1. Server looks up username -> if invalid, responds immediately
2. Server checks if account is locked
3. Server hashes provided password and compares with stored hash -> if invalid, responds after hashing delay

An attacker can measure response times:
- **Fast response** (no hashing) -> username does not exist
- **Slow response** (hashing executed) -> username exists, password wrong

**The Solution:**
1. **Constant-time comparison functions** — cryptographic functions that always take the same time regardless of input match
2. **Simulate artificial delays** — if username is not found, add a `sleep` or `setTimeout` to equalize response times

## 8. Key Takeaways

### Authentication Types
- **Stateful (Sessions):** Server stores session data, easy to revoke, harder to scale
- **Stateless (JWT):** Self-contained token, scales well, hard to revoke before expiry
- **API Keys:** Static secrets for machine-to-machine communication
- **OAuth 2.0 + OIDC:** Delegated authorization and identity via external providers

### Core Concepts
- **Authentication = Who are you?** (identity verification)
- **Authorization = What can you do?** (permission checking)
- **Sessions = Stateful server-side memory** using Session IDs and cookies
- **JWT = Stateless self-contained tokens** with header, payload, and signature
- **OAuth 2.0 = Delegated authorization** without password sharing
- **OpenID Connect = Authentication layer** on top of OAuth 2.0 (powers "Sign in with X")

### Security Best Practices
- Never store passwords in plain text — always **hash** them
- Return **generic error messages** for authentication failures
- Protect against **timing attacks** using constant-time comparisons or artificial delays
- Use **HttpOnly cookies** to prevent JavaScript from accessing session tokens
- For production, consider using a managed **Auth Provider** (Auth0, Clerk, etc.)

## 9. Glossary

| Term | Definition |
|------|------------|
| **Authentication** | The process of verifying who a user is. |
| **Authorization** | The process of determining what a user is allowed to do. |
| **Session** | A server-side mechanism to maintain state for a user across HTTP requests. |
| **Cookie** | A piece of data stored in the browser by the server, sent back with every request. |
| **JWT (JSON Web Token)** | A compact, self-contained token format used for stateless authentication. |
| **OAuth 2.0** | An authorization framework for delegated access to resources. |
| **OpenID Connect (OIDC)** | An authentication layer built on top of OAuth 2.0. |
| **RBAC** | Role-Based Access Control — assigning permissions to roles rather than individual users. |
| **Hashing** | A one-way cryptographic function that transforms data into a fixed-length string. |
| **Delegation** | Granting limited access to one's resources to another application without sharing passwords. |
| **MFA** | Multi-Factor Authentication — using multiple proof factors to verify identity. |
| **Timing Attack** | An attack that exploits time differences in system responses to infer information. |
| **API Key** | A static secret string used for programmatic server-to-server authentication. |
