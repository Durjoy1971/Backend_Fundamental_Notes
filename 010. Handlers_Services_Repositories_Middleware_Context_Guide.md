# Handlers, Services, Repositories, Middleware & Request Context: A Complete Guide

> **Prerequisites:** Client-server architecture, HTTP, REST APIs, JSON, serialization/deserialization, validations and transformations.

---

## 1. Request Lifecycle: Inside the Server

When a client sends an HTTP request to your server, a lot happens before a response is returned. This is the **request lifecycle inside the server**.

```
Client Request
      ↓
Server Entry Point (port 3000, 4000, etc.)
      ↓
Middleware Chain (optional)
      ↓
Routing (map URL to Handler)
      ↓
Middleware Chain (optional)
      ↓
Handler / Controller Layer
      ↓
Service Layer
      ↓
Repository Layer (Database)
      ↓
Service Layer (orchestration)
      ↓
Handler / Controller Layer
      ↓
Middleware Chain (optional)
      ↓
Response back to Client
```

**The flow:** Request reaches server → Middleware processes → Routing matches URL → Handler processes → Service executes business logic → Repository interacts with database → Response returns.

---

## 2. Three-Layer Architecture

A well-structured backend separates concerns into three layers:

| Layer | Responsibility | Example |
|-------|--------------|---------|
| **Handler / Controller** | HTTP logic: receive request, extract data, validate, transform, call service, decide response code | Receives JSON, returns 200 or 400 |
| **Service** | Business logic: orchestrate operations, call repositories, send emails, external API calls | Fetch user data, send notification |
| **Repository** | Database interaction: construct queries, execute CRUD operations | `SELECT * FROM books WHERE ...` |

**Why separate them?**
- **Scalability:** Each layer has a single responsibility
- **Maintainability:** Easier to debug and add features
- **Testability:** Can test business logic without HTTP concerns

> **Note:** This is a design pattern, not a hard requirement. You can put everything in one function, but it won't scale.

## 3. Handler / Controller Layer

The **Handler** (also called **Controller**) is the entry point for a specific API endpoint.

### What a Handler Receives

Depending on your programming language, each handler receives at least two objects:

| Object | Purpose |
|--------|---------|
| **Request** | Contains incoming data: headers, query params, path params, body |
| **Response** | Used to send response back: status code, headers, body |

These are provided by the runtime (Go, Node.js/Express, Python, etc.) — you don't create them yourself.

### Handler Responsibilities

#### 1. Extract Data from Request

Depending on the HTTP method:
- **GET:** Extract query parameters
- **POST/PUT/PATCH:** Extract request body
- **DELETE:** Extract request body (if any)

#### 2. Deserialize / Bind Data

The client sends data in JSON (serialized format). The server needs it in its native format.

| Language | Native Format |
|----------|--------------|
| Go | Struct |
| Python | Dictionary or Class |
| Node.js | JavaScript Object (often handled by middleware) |
| Rust | Struct |

This step is called **binding** — binding the request body to your native data structure.

If deserialization fails → Return **400 Bad Request** and terminate.

#### 3. Validate and Transform

We covered this in the Validations & Transformations guide:
- **Validate:** Check data format, types, constraints
- **Transform:** Set defaults, normalize (e.g., lowercase emails)

#### 4. Call Service Layer

Pass the validated and transformed data to the service layer. Also pass any additional context (user ID, permissions from authentication).

#### 5. Receive Data from Service

The service layer returns processed data (or an error).

#### 6. Send Response

Decide the appropriate HTTP status code:
- **Success:** 200 OK, 201 Created, 204 No Content
- **Client Error:** 400 Bad Request (validation failure)
- **Server Error:** 500 Internal Server Error

Send the response body with appropriate data or error message.

### Handler Flow Summary

```
1. Extract data from request object
2. Deserialize/bind to native format
3. Validate and transform
4. Call service layer with data
5. Receive result from service
6. Send response with appropriate status code
```

## 4. Service Layer

The **Service Layer** contains the actual business logic of your application.

### Key Principles

1. **No HTTP concerns:** Service methods should not know about HTTP, status codes, or request/response objects.
2. **Pure processing:** Take data, process it, return data.
3. **Orchestration:** Can call multiple repository methods, merge data, or call external services.

### Service Responsibilities

#### Orchestration

A service method can:
- Call multiple repository methods
- Merge data from different sources
- Transform data before returning

Example: Fetch user profile, fetch user's posts, merge into a single response.

#### External Operations

Services can perform operations that don't involve the database:
- Send emails
- Send notifications
- Call external APIs (webhooks, third-party services)
- Process files

#### Database Operations

If data needs to be persisted or fetched, the service calls the **Repository Layer**.

### Service Flow Summary

```
1. Receive data from Handler
2. Perform business logic
3. Call repository methods (if needed)
4. Orchestrate multiple operations
5. Return result to Handler
```

> **Rule:** One service method should do one logical thing. Don't mix unrelated operations.

## 5. Repository Layer

The **Repository Layer** (also called **Database Layer**) handles all direct database interactions.

### Key Principles

1. **Single Responsibility:** One repository method does one thing.
2. **No business logic:** Only construct and execute database queries.
3. **Return one type of data:** A method should return either a single item or a list, not both based on optional parameters.

### Repository Responsibilities

#### Construct Queries

Take data from the service layer and construct the appropriate database query.

Example:
```sql
-- Service passes: sort = "name"
SELECT * FROM books ORDER BY name;

-- Service passes: sort = "date"
SELECT * FROM books ORDER BY created_at;
```

#### Execute CRUD Operations

- **Create:** Insert new records
- **Read:** Fetch records (single or multiple)
- **Update:** Modify existing records
- **Delete:** Remove records

### Repository Design Anti-Pattern

**Don't do this:**
```javascript
// Bad: One method returns different types based on optional param
function getBook(id = null) {
  if (id) return singleBook;
  else return allBooks;
}
```

**Do this instead:**
```javascript
// Good: Separate methods for separate responsibilities
function getBookById(id) { return singleBook; }
function getAllBooks() { return allBooks; }
```

### Repository Flow Summary

```
1. Receive data from Service
2. Construct database query
3. Execute query
4. Return result to Service
```

## 6. Middleware

**Middleware** are functions that execute in the middle of the request lifecycle, before or after the handler.

### What is Middleware?

A middleware function receives three objects:

| Object | Purpose |
|--------|---------|
| **Request** | Access incoming data |
| **Response** | Send response back |
| **Next** | Pass execution to the next middleware or handler |

The `next` function is the key — it passes control to the next processing context.

### Where Middleware Executes

```
Request Entry
      ↓
[Middleware 1] → next()
      ↓
[Middleware 2] → next()
      ↓
Routing
      ↓
[Middleware 3] → next()
      ↓
Handler → Service → Repository
      ↓
[Middleware 4] → next()
      ↓
Response
```

### Why Use Middleware?

**To reduce code duplication.**

Without middleware, you would need to duplicate common logic in every handler:
- CORS checks
- Authentication
- Logging
- Rate limiting
- Error handling

With middleware, you write the logic once and it applies to all requests.

### Middleware Can:

- **Read** values from the request
- **Modify** the request or response
- **Send a response** immediately (terminating the request early)
- **Pass control** to the next middleware using `next()`

### Common Middleware Examples

#### 1. CORS (Cross-Origin Resource Sharing)

**Purpose:** Allow or block requests from specific origins (domains).

**How it works:**
1. Extract `Origin` header from request
2. Check if origin is in allowed list
3. If allowed, add CORS headers to response
4. If not allowed, don't add headers (browser will block response)

**Why middleware:** Needed for every request, deals with request/response objects.

#### 2. Security Headers

**Purpose:** Add security headers like Content-Security-Policy, X-Frame-Options.

**How it works:**
1. Receive request
2. Add security headers to response
3. Pass to next middleware

**Why middleware:** Needed for every response.

#### 3. Authentication

**Purpose:** Verify user identity before allowing access.

**How it works:**
1. Extract token from request (header, cookie, body)
2. Verify token (JWT, session ID)
3. If invalid → Return 401 Unauthorized immediately
4. If valid → Extract user ID, role, permissions
5. Store in **request context** (covered later)
6. Pass to next middleware

**Why middleware:** Needed for protected endpoints, can terminate early on failure.

#### 4. Rate Limiting

**Purpose:** Prevent clients from making too many requests in a short time.

**How it works:**
1. Extract client IP address
2. Check how many requests from this IP in the last X seconds
3. If over threshold → Return 429 Too Many Requests
4. If under threshold → Pass to next middleware

**Why middleware:** Needed for every request, can terminate early.

#### 5. Logging / Monitoring

**Purpose:** Log request details for debugging and auditing.

**How it works:**
1. Extract path, method, query params, body
2. Log to file or terminal
3. Pass to next middleware

**Why middleware:** Needed for every request.

#### 6. Global Error Handling

**Purpose:** Catch errors from anywhere in the application and return structured error responses.

**How it works:**
1. Placed as the **last middleware** (after all others)
2. Catches any error that occurred upstream
3. Determines if it's a client error (400) or server error (500)
4. Returns structured error message with code and message

**Why middleware:** Centralized error handling, accessible from any layer.

**Order matters:** Must be last to catch errors from all previous layers.

#### 7. Compression

**Purpose:** Compress large responses to reduce bandwidth.

**How it works:**
1. Check if response is large
2. Apply compression algorithm (gzip, etc.)
3. Add compression header to response
4. Browser decompresses on client side

**Why middleware:** Needed for large responses.

#### 8. Data Passing (Serialization/Deserialization)

**Purpose:** Handle JSON parsing and binding before handlers.

**How it works:**
1. Parse JSON body
2. Bind to native format
3. Pass to next middleware or handler

**Why middleware:** Common operation for POST/PUT requests.

### Middleware Order is Critical

The order determines how requests are processed:

```
1. CORS (early termination if origin not allowed)
2. Logging (log every request)
3. Authentication (verify before handler)
4. Rate Limiting (check after auth to track per-user)
5. Handler/Service/Repository
6. Global Error Handling (catch everything)
```

**Rule:** Place global error handling **last** so it can catch errors from all previous layers.

## 7. Request Context

**Request Context** is a shared state that is scoped to a single HTTP request. It is accessible across all middlewares and handlers.

### What is Request Context?

- A storage mechanism (typically key-value pairs)
- Scoped to one specific request
- Accessible by all middlewares and handlers
- Allows passing data without tightly coupling functions

### Why Use Request Context?

Without context, you would need to pass data explicitly through every function call, creating tight coupling.

With context, any middleware or handler can:
- **Set** a value in the context
- **Read** a value from the context
- No need to pass it as a parameter

### Common Use Cases

#### 1. User Information from Authentication

**Authentication middleware:**
1. Verifies token
2. Extracts user ID, role, permissions
3. Stores in request context

**Downstream handler:**
1. Reads user ID from context
2. Uses it to insert into database (e.g., `user_id` field in books table)
3. Checks role for authorization (admin vs user)

**Why context:** Avoids passing user ID through every function call. Also prevents malicious clients from spoofing user ID in request body.

#### 2. Request ID for Tracing

**Middleware:**
1. Generates unique UUID
2. Stores in request context

**Downstream uses:**
- Log the request ID in all log entries
- Pass request ID to external API calls (microservices)
- Trace request across multiple services for debugging

**Why context:** Enables distributed tracing without passing ID explicitly.

#### 3. Cancellation Signals and Deadlines

**Middleware:**
1. Sets timeout/deadline for request
2. Stores cancellation signal in context

**Downstream service:**
1. Checks context for cancellation
2. Aborts long-running operations if deadline exceeded

**Why context:** Prevents services from hanging indefinitely.

### Context Flow Example

```
[Auth Middleware]
  → Verify token
  → Set context.user_id = "123"
  → Set context.role = "admin"
  → next()

[Handler]
  → Read context.user_id
  → Use in database insert
  → Read context.role
  → Check authorization
  → next()

[Logging Middleware]
  → Read context.request_id
  → Log with request ID
  → next()
```

## 8. Complete Request Lifecycle Example

Let's trace a complete request through all layers:

**Request:** `POST /api/books` with JSON body containing book data.

```
1. Request reaches server entry point (port 3000)

2. [CORS Middleware]
   - Check Origin header
   - Add CORS headers to response
   - next()

3. [Logging Middleware]
   - Log path, method, body
   - Generate request_id, store in context
   - next()

4. [Authentication Middleware]
   - Extract JWT from Authorization header
   - Verify signature
   - Extract user_id, role
   - Store in context
   - next()

5. [Routing]
   - Match POST /api/books to createBookHandler

6. [Handler: createBookHandler]
   - Extract request body
   - Deserialize JSON to Book struct
   - Validate: title required, author required
   - Transform: set default status = "draft"
   - Call service: createBook(bookData, context.user_id)

7. [Service: createBook]
   - Receive bookData and user_id
   - Call repository: insertBook(bookData, user_id)
   - Call external service: sendNotification(user_id, "Book created")
   - Return created book to handler

8. [Repository: insertBook]
   - Construct SQL: INSERT INTO books (title, author, user_id) VALUES (...)
   - Execute query
   - Return inserted book to service

9. [Service]
   - Receive book from repository
   - Return to handler

10. [Handler]
    - Receive book from service
    - Serialize to JSON
    - Set status code: 201 Created
    - Send response

11. [Global Error Handling Middleware]
    - No error occurred
    - Pass response through

12. Response sent to client
```

## 9. Key Takeaways

### Three-Layer Architecture
- **Handler/Controller:** HTTP logic, validation, response codes
- **Service:** Business logic, orchestration, external calls
- **Repository:** Database operations, query construction

### Middleware
- Functions that execute in the middle of request lifecycle
- Receive request, response, and `next` function
- Reduce code duplication for common operations
- Order is critical (CORS → Logging → Auth → Handler → Error Handling)

### Request Context
- Shared state scoped to a single request
- Accessible across all middlewares and handlers
- Used for user info, request IDs, cancellation signals
- Avoids tight coupling between functions

### Design Principles
- Separation of concerns makes code scalable and maintainable
- Each layer has a single responsibility
- Validate early, fail fast with appropriate status codes
- Use middleware for cross-cutting concerns
- Use context for request-scoped shared state

## 10. Glossary

| Term | Definition |
|------|------------|
| **Handler / Controller** | The function that handles a specific API endpoint, deals with HTTP concerns |
| **Service Layer** | Contains business logic, orchestrates operations, calls repositories |
| **Repository Layer** | Handles database interactions, constructs and executes queries |
| **Middleware** | Functions that execute in the middle of the request lifecycle |
| **Next Function** | The function that passes control to the next middleware or handler |
| **Request Context** | Shared state scoped to a single request, accessible across middlewares and handlers |
| **Binding** | Converting request data (JSON) to native language format (struct, class, dictionary) |
| **Orchestration** | Coordinating multiple operations (e.g., calling multiple repositories, merging data) |
| **CORS** | Cross-Origin Resource Sharing — security mechanism for cross-domain requests |
| **Rate Limiting** | Limiting the number of requests a client can make in a time period |
| **Global Error Handling** | Centralized error catching and structured error response generation |
| **Request ID** | Unique identifier for a request, used for tracing and logging |
| **Cancellation Signal** | Signal to abort long-running operations when a deadline is exceeded |
