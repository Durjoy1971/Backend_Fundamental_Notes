# REST API Design: A Complete Guide

> **Prerequisites:** HTTP, client-server architecture, JSON, serialization/deserialization.

---

## 1. Introduction & History

### The Origins of the Web

In 1990, **Tim Berners-Lee** started the World Wide Web project to share knowledge globally. Within a year, he invented:
- **URI** (Uniform Resource Identifier)
- **HTTP** (the protocol for client-server communication)
- **HTML** (markup language for web pages)
- The first web server, browser, and HTML editor

These technologies are still in use today, though in more advanced forms (HTTP 2.0, 3.0, modern browsers, etc.).

### The Scalability Crisis

The web grew exponentially, and the original architecture couldn't handle the massive user base. Around 1993, **Roy Fielding** (co-founder of the Apache HTTP Server project) proposed constraints to solve this scalability problem.

### REST Constraints

Fielding proposed six constraints to make the web more scalable:

| Constraint | Description |
|------------|-------------|
| **Client-Server** | Separation of concerns: client handles UI/UX, server manages data and business logic |
| **Uniform Interface** | Standardized way for components to communicate (includes resource identification, manipulation through representation, self-descriptive messages, HATEOAS) |
| **Layered System** | Hierarchical layers where each layer only interacts with the immediate layer below (enables load balancers, proxies) |
| **Cacheable** | Responses must be explicitly labeled as cacheable or non-cacheable to reduce server load |
| **Stateless** | Each request must contain all information needed to process it; server doesn't remember previous requests |
| **Code on Demand** (Optional) | Servers can temporarily extend client functionality by transferring executable code (e.g., JavaScript) |

Fielding and Berners-Lee collaborated to write the specification for **HTTP 1.1**, the first standard version of HTTP.

### What is REST?

In 2000, Fielding named and described the web's architectural style in his PhD dissertation as **REST** (Representational State Transfer).

The name breaks down into three parts:

1. **Representational**: Resources are represented in specific formats (JSON, XML, HTML). The same resource can have different representations for different clients.
2. **State**: The current condition or attributes of a resource (e.g., a shopping cart with items, quantities, and total price).
3. **Transfer**: Movement of resource representations between client and server via HTTP methods (GET, POST, PUT, DELETE, PATCH, etc.).

**REST describes an architectural style where:**
- Resources have different formats (representations)
- The state of resources can be transferred between client and server
- The system follows specific constraints for scalability

## 2. URL Structure

### URL Components

A typical URL structure:

```
scheme://authority/path?query#fragment
```

| Component | Example | Description |
|-----------|---------|-------------|
| **Scheme** | `https://` | Protocol (HTTP or HTTPS) |
| **Authority** | `api.example.com` | Domain (can include subdomain) |
| **Path** | `/v1/books` | Resource being accessed |
| **Query** | `?page=1&limit=10` | Key-value pairs for filters, pagination |
| **Fragment** | `#section` | Navigates to a specific section of a page |

### API URL Structure

For REST APIs, the industry standard follows this pattern:

```
https://api.example.com/v1/books
```

| Part | Value | Purpose |
|------|-------|---------|
| **Scheme** | `https://` | Secure protocol |
| **Subdomain** | `api` | Separates API from main domain |
| **Domain** | `example.com` | Main domain |
| **Version** | `/v1` | API versioning (v1, v2, etc.) |
| **Resource** | `/books` | Plural form of the resource |

### Path Segments and Hierarchical Relationships

The forward slash (`/`) represents a **hierarchical relationship** between resources.

```
/books              → Collection of all books
/books/{id}         → A specific book from the collection
/books/{id}/reviews → Reviews for a specific book
```

### URL Naming Rules

- **Always use lowercase** for path segments
- **Use hyphens (-) for multi-word slugs**, not spaces or underscores
- **Resources should always be plural** in path segments

**Example:**
- Book name: "Harry Potter"
- Slug: `harry-potter`
- URL: `/books/harry-potter`

> **Rule:** Even when fetching a single resource, use the plural form. `/books/{id}` not `/book/{id}`.

## 3. HTTP Methods & Idempotency

### HTTP Methods

The five major HTTP methods used in REST APIs:

| Method | Purpose | Idempotent |
|--------|---------|------------|
| **GET** | Retrieve data | Yes |
| **POST** | Create resource or custom action | No |
| **PUT** | Completely replace a resource | Yes |
| **PATCH** | Partially update a resource | Yes |
| **DELETE** | Delete a resource | Yes |

### What is Idempotency?

**Idempotency** means that performing the same action multiple times has the same effect as performing it once.

- **Idempotent methods:** GET, PUT, PATCH, DELETE
- **Non-idempotent method:** POST

### Method Details

#### GET
- Used to retrieve data
- **Idempotent**: Calling it multiple times doesn't change server state
- Example: Fetch list of books, fetch single book

#### POST
- Used to create a new resource OR for custom actions
- **Non-idempotent**: Each call creates a new resource
- Example: Create a book, send an email, archive an organization

#### PUT
- Used to **completely replace** a resource representation
- **Idempotent**: Calling it multiple times with same payload has same effect
- Example: Replace entire user object (all fields required)

#### PATCH
- Used to **partially update** a resource
- **Idempotent**: Calling it multiple times with same payload has same effect
- Example: Update only the name field of a user
- **Best practice**: Use PATCH for partial updates in modern APIs

#### DELETE
- Used to delete a resource
- **Idempotent**: First call deletes, subsequent calls return 404 (no further state change)
- Example: Delete a book

### Custom Actions with POST

When an action doesn't fit into CRUD (Create, Read, Update, Delete), use POST:

- **Send email**: `POST /send-email`
- **Archive organization**: `POST /organizations/{id}/archive`
- **Clone project**: `POST /projects/{id}/clone`

POST is "open-ended" in the REST spec for such scenarios.

## 4. API Design Workflow

### Step 1: Start with UI/UX

Begin with wireframes, Figma designs, or user stories. This gives you insight into:
- How end users interact with data
- The relationship between users and your database
- What resources are needed

### Step 2: Identify Resources

From requirements/wireframes, extract **nouns** — these are your resources.

**Example (Project Management Platform):**
- Users
- Organizations
- Projects
- Tasks
- Tags

### Step 3: Database Schema Design

Design your database tables based on resources. (This is covered in a separate video on databases.)

### Step 4: API Interface Design

Design the API interface **before coding**. Use tools like:
- **Insomnia** or **Postman** for API client testing
- **Swagger/OpenAPI** for interactive documentation

> **Important:** Design the interface first, then implement. Don't jump straight into coding.

## 5. CRUD Operations

### Create (POST)

**Route:** `POST /organizations`

**Request Body:**
```json
{
  "name": "Org One",
  "status": "active",
  "description": "Some description"
}
```

**Response:**
- **Status Code:** `201 Created`
- **Body:** The newly created entity with server-generated fields (id, createdAt, updatedAt)

**Note:** Exclude server-handled fields (id, createdAt, updatedAt) from the request body.

### Read (GET)

#### List All Resources
**Route:** `GET /organizations`

**Response:**
- **Status Code:** `200 OK`
- **Body:** Array of resources (or paginated response)

#### Get Single Resource
**Route:** `GET /organizations/{id}`

**Response:**
- **Status Code:** `200 OK` if found
- **Status Code:** `404 Not Found` if resource doesn't exist
- **Body:** Single resource object

### Update (PATCH)

**Route:** `PATCH /organizations/{id}`

**Request Body:**
```json
{
  "status": "active"
}
```

**Response:**
- **Status Code:** `200 OK`
- **Body:** The updated entity

**Note:** Send only the fields you want to update (partial update).

### Delete (DELETE)

**Route:** `DELETE /organizations/{id}`

**Response:**
- **Status Code:** `204 No Content`
- **Body:** Empty (no content to return)

**Note:** 204 indicates success but no content to return.

## 6. Pagination

### Why Paginate?

1. **Performance:** Serializing thousands of records into JSON is resource-heavy
2. **User Experience:** Users typically only see 10-20 items at a time
3. **Network Efficiency:** Reduces bandwidth usage

### Pagination Response Structure

```json
{
  "data": [...],        // The portion of data for this page
  "total": 50,          // Total count of all resources
  "page": 1,           // Current page number
  "totalPages": 5      // Total number of pages
}
```

### Pagination Parameters

| Parameter | Purpose | Default |
|-----------|---------|---------|
| `limit` | Number of items per page | 10 or 20 |
| `page` | Which portion of data to return | 1 |

**Example Request:**
```
GET /organizations?limit=2&page=1
```

**Example Response:**
```json
{
  "data": [
    {"id": "5", "name": "Org 5"},
    {"id": "4", "name": "Org 4"}
  ],
  "total": 5,
  "page": 1,
  "totalPages": 3
}
```

### Sensible Defaults

Always provide defaults:
- If `page` not sent → default to `1`
- If `limit` not sent → default to `10` or `20`

This ensures the API works even without explicit parameters.

## 7. Sorting

### Sorting Parameters

| Parameter | Purpose | Default |
|-----------|---------|---------|
| `sortBy` | Field to sort by | `createdAt` |
| `sortOrder` | Order direction | `descending` |

**Example Request:**
```
GET /organizations?sortBy=name&sortOrder=ascending
```

### Default Sorting

Always sort by default to ensure consistent responses:
- **Default field:** `createdAt`
- **Default order:** `descending` (latest first)

Without default sorting, the database returns data in random order, causing inconsistent responses.

---

## 8. Filtering

### Filtering by Fields

Clients can filter lists by specific fields.

**Example Request:**
```
GET /organizations?status=active
```

**Example Request:**
```
GET /organizations?name=org
```

### Empty Results in List APIs

**Important Rule:** For list APIs, return an empty array with `200 OK`, **not** `404`.

**Correct Response (no matches):**
```json
{
  "data": [],
  "total": 0,
  "page": 1,
  "totalPages": 0
}
```
Status Code: `200 OK`

**When to use 404:** Only when requesting a **specific resource by ID** that doesn't exist.

## 9. Custom Actions

### What are Custom Actions?

Actions that don't fit into CRUD operations:
- Archive an organization (involves deleting projects, notifying users)
- Clone a project (copies tasks, sends notifications)
- Send an email

### Custom Action Route Structure

```
POST /organizations/{id}/archive
POST /projects/{id}/clone
POST /send-email
```

**Pattern:** `POST /{resource}/{id}/{action}`

### Why Not Use PATCH?

Even though archiving changes the status field, it's a custom action because:
- Multiple server-side operations may be needed
- Business logic beyond simple field update
- Notifications, cascading deletes, etc.

### Status Codes for Custom Actions

- `200 OK` if no resource created (e.g., archive)
- `201 Created` if a resource is created (e.g., clone creates new project)

Don't assume POST always returns 201.

## 10. Status Codes

| Status Code | Meaning | Use Case |
|------------|---------|----------|
| **200 OK** | Success | GET, PATCH, custom actions that don't create |
| **201 Created** | Resource created | POST create operations |
| **204 No Content** | Success, no content | DELETE operations |
| **400 Bad Request** | Invalid input | Validation failures |
| **404 Not Found** | Resource not found | GET/PATCH/DELETE for specific ID that doesn't exist |
| **429 Too Many Requests** | Rate limit exceeded | Client made too many requests |

### 404 Rule

- **Use 404** when requesting a specific resource by ID that doesn't exist
- **Don't use 404** for list APIs with no results → return empty array with 200

## 11. Consistency Principles

### Consistent Routes

Once you establish a pattern, stick to it across all resources.

**Example:**
- `/organizations` (plural) → `/projects` (plural)
- `/organizations/{id}` → `/projects/{id}`

If you use singular for one resource, clients will assume the same for others. Inconsistency causes confusion and wasted time.

### Consistent Payloads

Field names should be consistent across resources.

**Example:**
- If `description` is used in `/organizations`, use `description` in `/projects`
- Don't use `desc` or `dsc` — be consistent

### JSON Standards

All JSON fields should follow **camelCase**:

```json
{
  "organizationId": "123",
  "createdAt": "2024-01-01",
  "isActive": true
}
```

This is the established JSON standard. Don't use snake_case or PascalCase.

### Avoid Abbreviations

Don't use abbreviations like `desc` for `description` or `org` for `organization`.

**Why?** Consumers of your API don't have the same context you do. Abbreviations cause confusion and errors.

## 12. Sensible Defaults

### Defaults in GET Requests

| Parameter | Default Value |
|-----------|---------------|
| `page` | `1` |
| `limit` | `10` or `20` |
| `sortBy` | `createdAt` |
| `sortOrder` | `descending` |

### Defaults in POST Requests

Only require fields that are absolutely necessary. Provide safe defaults for others.

**Example:**
```json
{
  "name": "Org One",
  "description": "Some description"
}
```

If `status` is not sent, default to `active` (safe assumption based on business logic).

### Why Defaults Matter

- Better user experience (fewer required fields)
- Less guesswork for API consumers
- More intuitive API

## 13. Documentation & Design-First Approach

### Interactive Documentation

Use tools like **Swagger/OpenAPI** to provide:
- Interactive playground for testing APIs
- Form of documentation
- Form of testing for consumers

**Important:** Plan documentation from the initial design phase. Consistently maintain and update it.

### Design Before Coding

**Workflow:**
1. Start with wireframes/Figma
2. Identify resources
3. Design database schema
4. **Design API interface** (using Insomnia, Postman, or Swagger)
5. Then implement in code

**Why?**
- Designing first gives insights into how consumers will use your API
- Forces you to think about the interface without language/framework bias
- Results in more intuitive APIs

### API Clients

Use tools like **Insomnia** or **Postman** to design and test your API interface before writing any backend code.

## 14. Key Takeaways

### REST Fundamentals
- REST is an architectural style for scalable web applications
- Resources are represented in different formats (JSON, XML, HTML)
- State is transferred between client and server via HTTP methods

### URL Design
- Use lowercase, hyphens for multi-word slugs
- Resources should always be plural in path segments
- Forward slash represents hierarchical relationships

### HTTP Methods
- GET (retrieve), POST (create/custom action), PUT (replace), PATCH (partial update), DELETE (delete)
- POST is the only non-idempotent method
- Use POST for custom actions that don't fit CRUD

### CRUD Operations
- Create: POST with 201 status
- Read: GET with 200 (list) or 200/404 (single)
- Update: PATCH with 200
- Delete: DELETE with 204

### List API Features
- Pagination: limit, page parameters with defaults
- Sorting: sortBy, sortOrder with defaults
- Filtering: filter by fields
- Empty results: return empty array with 200, not 404

### Consistency
- Consistent routes, payloads, responses across all resources
- Use camelCase for JSON fields
- Avoid abbreviations
- Provide sensible defaults

### Design-First
- Design API interface before coding
- Use Swagger/OpenAPI for documentation
- Use Insomnia/Postman for testing during design phase

| Term | Definition |
|------|------------|
| **REST** | Representational State Transfer — architectural style for web APIs |
| **Idempotency** | Property where performing an action multiple times has the same effect as once |
