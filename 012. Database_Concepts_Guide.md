## 1. Introduction to Databases

### Why Do We Need Databases?

At its core, a database is simply a way to **persist information** across different sessions. Persistence means storing data so that it survives even after the program that created it has stopped.

**Real-world example:** Think about a to-do list app on your phone. You add tasks, check them off, and close the app. When you open it again later, all your tasks are still there - that's persistence in action! Without persistence, every time you opened the app, you'd have to start with an empty list.

### What Is a Database?

The term "database" is surprisingly broad. In the simplest sense, **any structured storage system** can be considered a database:

- Your phone's contact list
- Browser's local storage (key-value store)
- A simple text file with notes
- A spreadsheet

All these systems provide a way to **Create, Read, Update, and Delete (CRUD)** data. That's the fundamental operation pattern of any database.

However, when backend engineers talk about databases, they typically mean **disk-based databases** - software systems that store data on hard drives or SSDs rather than in RAM.

## 2. Storage Fundamentals

### Disk-Based vs RAM-Based Storage

Computers have two main types of memory:

| Storage Type | Speed | Capacity | Cost |
|--------------|-------|----------|------|
| RAM (Primary Memory) | Very Fast | Limited (8-128 GB typical) | Expensive |
| Disk (Secondary Memory) | Slower | Large (512 GB - 2 TB typical) | Cheap |

**Why do databases use disk storage?**

- **Capacity:** You can store much more data on disk than in RAM
- **Cost:** Disk storage is significantly cheaper per GB
- **Persistence:** Disk storage retains data even when power is lost

The trade-off is speed - RAM is much faster for data access. This is why **caching** technologies like Redis exist - they store frequently accessed data in RAM for fast retrieval, while the primary database uses disk storage for capacity and cost efficiency.

**When to use each:**

- **Use RAM-based storage (caching):** For frequently accessed data that needs instant retrieval
- **Use disk-based storage (databases):** For long-term data storage where capacity matters more than speed

## 3. Database Management Systems (DBMS)

### What Is a DBMS?

Just storing data on disk isn't enough. We need software to efficiently manage that data - this is where **Database Management Systems (DBMS)** come in.

A DBMS is software whose responsibility is to:
1. Store data efficiently
2. Provide CRUD operations (Create, Read, Update, Delete)
3. Handle security, scaling, and load balancing

### Key Responsibilities of a DBMS

1. **Data Organization:** Structuring data so that fetching, updating, and creating operations are efficient
2. **Access Control:** Providing methods for CRUD operations
3. **Integrity:** Ensuring data accuracy and validity
4. **Security:** Protecting data from unauthorized access through user roles and permissions

### Why Not Just Use Text Files?

Before DBMS existed, people tried storing data in plain text files. This had several problems:

**Problem 1: Parsing**
- Every time you need to find specific data, you must write code to parse the entire file
- This is slow and error-prone
- Performance varies greatly between programming languages (Rust is fast, Python/JavaScript slower)

**Problem 2: No Structure**
- Text files have no formal structure
- You can't enforce that a field must be a number or follow a specific format
- Data consistency is difficult to maintain

**Problem 3: Concurrency Issues**
- What happens when two users try to update the same data simultaneously?
- Without proper concurrency control, one update might overwrite the other
- Text files have no mechanism to handle this

**Example of concurrency problem:**
```
Two users read the same value: 40
User A increases it by 20 → saves 60
User B decreases it by 20 → saves 20
Final value: Either 60 or 20 (depending on who saved last)
Correct value should be: 40 (40 + 20 - 20)
```

A DBMS handles all these problems efficiently, which is why we use them instead of raw files.

## 4. Types of Databases

On a high level, there are two major types of DBMS:

### Relational Databases

**Characteristics:**
- Data organized in **tables, rows, and columns**
- Relationships between tables defined using **foreign keys**
- **Strict schema** - you must define columns and data types before inserting data
- Uses **SQL (Structured Query Language)** for queries

**Examples:** PostgreSQL, MySQL, SQL Server, SQLite

**Key advantage:** Data integrity - you can always trust the structure and type of your data

### Non-Relational (NoSQL) Databases

**Characteristics:**
- **Flexible schema** - no predefined structure required
- Data stored as **documents** (in MongoDB) or other formats
- Each document can have different structure
- No strict relationships enforced

**Examples:** MongoDB, Cassandra, Redis, DynamoDB

**Key advantage:** Flexibility - you can store any data structure without planning ahead

### Choosing Between Them

**Use Relational (e.g., PostgreSQL) when:**
- Data structure is well-defined and consistent
- Data integrity is critical (e.g., financial data, customer relationships)
- You need complex queries and relationships
- Example: **CRM (Customer Relationship Management)** - needs accurate, consistent customer data with complex relationships

**Use Non-Relational (e.g., MongoDB) when:**
- Data structure is unpredictable or evolving
- You need to move fast during prototyping
- Data doesn't have strict relationships
- Example: **CMS (Content Management System)** - articles can have images, code blocks, videos, embeddings - very flexible content

**Trade-off:** Non-relational databases offer flexibility but can introduce data integrity issues that you must handle in application code (more error-prone).

## 6. PostgreSQL Data Types

PostgreSQL offers a rich set of data types. Let's explore the most important ones for backend engineering.

### Numeric Types

**Integer Types:**
- `smallint` - Smallest range
- `integer` - Standard integer
- `bigint` - Largest range
- `serial` / `bigserial` - Auto-incrementing integers (great for IDs)

**Decimal vs Floating Point:**

| Type | Accuracy | Speed | Use Case |
|------|----------|-------|----------|
| `decimal` / `numeric` | Always accurate | Slower | Price, financial data (accuracy matters) |
| `real` / `double precision` | Can have small discrepancies | Faster | Scientific calculations, sizes (speed matters) |

**Rule of thumb:** Use `decimal` when accuracy is critical (prices, money). Use floating point when small discrepancies are acceptable and performance matters.

**Example with decimal precision:**
```sql
price decimal(10,2)
-- 10 = maximum total digits
-- 2 = digits after decimal point
-- Can store: 12345678.90 (10 digits total, 2 after decimal)
```

### String Types

**Three options:**
1. `char(n)` - Fixed length, pads with spaces if shorter (old standard, rarely used)
2. `varchar(n)` - Variable length, maximum of n characters
3. `text` - Variable length, no maximum limit (recommended)

**Recommendation:** Always use `text` instead of `varchar(n)`. Reasons:
- No performance difference in PostgreSQL
- Avoids arbitrary length limits (like 255) that have no meaning
- No need for database migrations to increase length
- Enforce length in application code instead

**When to use `char`:** Only when you know all values have the exact same length (e.g., day codes: 'MO', 'TU', 'WE')

### Boolean, Date, and Time

- `boolean` - true or false
- `date` - Date only (no time)
- `time` - Time only (hour, minute, second)
- `timestamp` - Date and time
- `timestamp with time zone` - Date, time, and timezone information
- `interval` - Duration (e.g., "10 days", "1 week")

### UUID

Universally Unique Identifier - excellent for primary keys:
- Globally unique (no collisions)
- Not sequential (harder to guess)
- PostgreSQL has native UUID type

### JSON and JSONB

PostgreSQL has excellent JSON support:

- `json` - Stored as plain text (slower)
- `jsonb` - Stored in binary format (faster, recommended)

**Use JSONB when:**
- Storing flexible/dynamic data
- Content that doesn't have a strict schema
- Need to query within JSON fields

### Array Types

You can store arrays of any data type:
```sql
numbers integer[]
tags text[]
dates date[]
```

### Other Specialized Types

- Network addresses (IP, MAC)
- Geometric points
- XML
- And many more (explore docs if needed)

## 7. Database Migrations

### What Are Migrations?

In production systems, you don't just open a GUI tool and execute SQL queries. Instead, you use **migrations** - files that track all changes to your database schema over time.

**Migration structure:**
```
db/
  migrations/
    1_create_users_table.sql
    2_add_email_index.sql
    3_seed_initial_data.sql
```

### Up vs Down Migrations

Each migration file has two sections:

**Up Migration:**
- Changes you want to apply (CREATE TABLE, ADD COLUMN, CREATE INDEX, etc.)
- Applied when moving forward in version history

**Down Migration:**
- Reverses the up migration (DROP TABLE, DROP COLUMN, DROP INDEX, etc.)
- Applied when rolling back to a previous version

**Example:**
```sql
-- Up Migration
CREATE TYPE project_status AS ENUM ('active', 'completed', 'archived');
CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  status project_status NOT NULL DEFAULT 'active'
);

-- Down Migration
DROP TABLE projects;
DROP TYPE project_status;
```

### Why Use Migrations?

1. **Track changes:** All database changes are versioned and committed to git
2. **Rollback capability:** If something breaks, you can revert to a previous state
3. **Team collaboration:** Everyone can see the history of schema changes
4. **Consistency:** Ensures all environments (dev, staging, prod) have the same schema

### Migration Tools

Popular migration tools:
- **dbmate** (used in examples)
- **go-migrate**
- **Flyway**
- **Alembic** (Python)

These tools:
- Track which migrations have been applied (usually in a `schema_migrations` table)
- Apply migrations in sequence
- Handle up and down migrations

## 8. Database Modeling

Let's design a database for a **project management platform** to learn modeling concepts.

### Naming Conventions

**Table names:**
- Use **plural form** (industry standard): `users`, `projects`, `tasks`
- Some teams prefer singular - choose what your team agrees on

**Column names:**
- Always **lowercase**
- Use **snake_case** for multi-word names: `full_name`, `password_hash`, `created_at`
- PostgreSQL is case-insensitive by default, so camelCase requires quotes (ugly in code)

### Default Fields

Most tables should have these metadata fields:

```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
```

- `id`: Unique identifier for each row (primary key)
- `created_at`: When the row was first created
- `updated_at`: When the row was last modified

### Enums

Enums are custom types with a predefined set of allowed values.

**Example:**
```sql
CREATE TYPE project_status AS ENUM ('active', 'completed', 'archived');
CREATE TYPE task_status AS ENUM ('pending', 'in_progress', 'completed', 'cancelled');
CREATE TYPE member_role AS ENUM ('owner', 'admin', 'member');

CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  status project_status NOT NULL DEFAULT 'active'
);
```

**Benefits of enums:**
1. **Data Integrity:** Database rejects invalid values (no need to check in app code)
2. **Documentation:** At a glance, anyone can see allowed values
3. **Self-documenting schema:** Future team members understand constraints without reading application code

## 9. Table Relationships

### One-to-One Relationship

Each record in Table A relates to exactly one record in Table B.

**Example:** Users and User Profiles

```sql
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT NOT NULL UNIQUE,
  full_name TEXT NOT NULL,
  password_hash TEXT NOT NULL
);

CREATE TABLE user_profiles (
  user_id UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  image_url TEXT,
  bio TEXT,
  phone_number TEXT
);
```

**Key points:**
- `user_id` is both the primary key AND foreign key in `user_profiles`
- No separate `id` field needed in profiles table
- Each user has exactly one profile (or none)

**Why separate tables?**
- User profile data can grow (social media, websites, projects)
- Frequent profile changes don't affect the core users table
- Keeps the main users table clean and stable

### One-to-Many Relationship

Each record in Table A can relate to multiple records in Table B.

**Example:** Projects and Tasks

```sql
CREATE TABLE projects (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  owner_id UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT
);

CREATE TABLE tasks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  status task_status NOT NULL DEFAULT 'pending'
);
```

**Key points:**
- `project_id` is a foreign key (not primary key) in tasks table
- One project can have many tasks
- One task belongs to exactly one project

### Many-to-Many Relationship

Records in Table A can relate to multiple records in Table B, and vice versa.

**Example:** Users and Projects (a user can be in many projects, a project can have many users)

```sql
CREATE TABLE project_members (
  project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role member_role NOT NULL DEFAULT 'member',
  PRIMARY KEY (project_id, user_id)
);
```

**Key points:**
- Requires a **linking table** (also called junction table)
- Composite primary key: `(project_id, user_id)` together uniquely identify each row
- Ensures a user can't be added to the same project twice
- Both foreign keys have `ON DELETE CASCADE` - removing a user or project removes their membership entries

## 10. Constraints

Constraints are rules that enforce data integrity at the database level.

### Primary Key

- Uniquely identifies each row
- Implicitly includes: `NOT NULL` and `UNIQUE` constraints
- Automatically indexed by the database

```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid()
```

### Not Null

- Field cannot contain NULL values
- Most fields should have this constraint
- Prevents accidental NULL values from bugs or scripts

```sql
email TEXT NOT NULL
```

**Rule:** Apply `NOT NULL` to 70%+ of your fields unless there's a specific reason to allow NULL.

### Unique

- All values in this column must be different
- Great for fields that should never duplicate

```sql
email TEXT NOT NULL UNIQUE  -- No two users can have the same email
```

### Check

- Custom condition that must be true
- Validates data beyond simple type checking

```sql
priority INTEGER NOT NULL DEFAULT 1 CHECK (priority >= 1 AND priority <= 5)
-- Priority must be between 1 and 5
```

### Foreign Key and Referential Integrity

Foreign keys establish relationships between tables and enforce referential integrity.

**Referential Integrity Options (ON DELETE):**

| Option | Behavior |
|--------|----------|
| `RESTRICT` | Prevents deletion if related records exist (default) |
| `CASCADE` | Deletes related records automatically |
| `SET NULL` | Sets foreign key to NULL (if field allows NULL) |
| `SET DEFAULT` | Sets foreign key to default value |

**Examples:**

```sql
-- Prevent deleting a user if they own projects
owner_id UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT

-- Delete tasks when project is deleted
project_id UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE

-- Unassign task if assigned user is deleted
assigned_to UUID REFERENCES users(id) ON DELETE SET NULL
```

## 11. Seeding

### What Is Seeding?

**Seeding** means populating your database with test data for development and testing purposes.

**Why seed?**
- Test your application with realistic data
- Verify queries work as expected
- Develop features without waiting for real user data
- Demo the application to stakeholders

### How to Seed

Create a separate migration file for seeding:

```sql
-- Up Migration (seed data)
INSERT INTO users (email, full_name, password_hash)
VALUES 
  ('alice@example.com', 'Alice Johnson', 'hashed_password_1'),
  ('bob@example.com', 'Bob Smith', 'hashed_password_2');

INSERT INTO user_profiles (user_id, bio, phone_number)
SELECT id, 'Software developer', '555-0101'
FROM users
WHERE email = 'alice@example.com';

-- Down Migration (remove seed data)
DELETE FROM user_profiles;
DELETE FROM users;
```

**Best practice:** Keep seeding in a separate migration file from schema changes.

## 12. SQL Query Fundamentals

### Joins

Joins combine data from multiple tables based on related columns.

**INNER JOIN:** Returns rows when there's a match in BOTH tables

```sql
SELECT u.*, p.*
FROM users u
INNER JOIN user_profiles p ON u.id = p.user_id;
-- Only returns users who HAVE profiles
```

**LEFT JOIN:** Returns ALL rows from left table, matching rows from right table (NULL if no match)

```sql
SELECT u.*, row_to_jsonb(p.*) as profile
FROM users u
LEFT JOIN user_profiles p ON u.id = p.user_id;
-- Returns ALL users, with profile data if it exists
```

**When to use which:**
- Use `INNER JOIN` when both sides must exist
- Use `LEFT JOIN` when you want the primary table's data regardless of whether the related data exists

### Parameterized Queries

Parameterized queries protect against **SQL injection** attacks.

**The problem with string concatenation:**
```sql
-- DANGEROUS - vulnerable to SQL injection
query = "SELECT * FROM users WHERE id = '" + user_input + "'";
-- If user_input is "'; DROP TABLE users; --" - disaster!
```

**The solution - parameterized queries:**
```sql
-- SAFE - user input is treated as a string, not executable code
SELECT * FROM users WHERE id = $1;
-- The database fills in $1 with the actual value
```

**How it works:**
1. You provide a query with "slots" (parameters)
2. You tell the database: "Here's my query, I'll fill in the slots later"
3. When you provide values, they're treated as literal strings (escaped)
4. No SQL injection possible

**In your application code**, your database driver/ORM handles parameters automatically.

### Common Table Expressions (CTEs)

CTEs create temporary result sets that can be referenced within a query.

```sql
WITH inserted_users AS (
  INSERT INTO users (email, full_name, password_hash)
  VALUES ('test@example.com', 'Test User', 'hash')
  RETURNING id, email
)
INSERT INTO user_profiles (user_id, bio)
SELECT id, 'New user bio'
FROM inserted_users;
```

**Benefits:**
- Makes complex queries more readable
- Allows you to reference the result of one operation in another
- Great for multi-step operations

### Converting Rows to JSON

PostgreSQL can convert entire rows to JSON:

```sql
SELECT 
  u.*,
  row_to_jsonb(up.*) as profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id;
```

**Result:**
```json
{
  "id": "uuid-1",
  "email": "alice@example.com",
  "full_name": "Alice Johnson",
  "profile": {
    "user_id": "uuid-1",
    "bio": "Software developer",
    "phone_number": "555-0101"
  }
}
```

This is perfect for APIs where you want to embed related data in a single response.

## 13. Dynamic Query Features

### Dynamic Filtering

Users often want to filter results by various criteria.

**Example: Filter users by first letter of name**
```sql
SELECT u.*, row_to_jsonb(up.*) as profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
WHERE u.full_name ILIKE $1 || '%';
-- $1 is the letter parameter
-- ILIKE is case-insensitive pattern matching
-- 'J%' matches names starting with 'J'
```

**Pattern matching with LIKE:**
- `%` matches any sequence of characters
- `_` matches any single character
- `ILIKE` is case-insensitive, `LIKE` is case-sensitive

### Dynamic Sorting

Allow users to choose sort field and direction:

```sql
SELECT u.*, row_to_jsonb(up.*) as profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
ORDER BY $1 $2;
-- $1 = sort field (e.g., "u.email", "u.full_name", "u.created_at")
-- $2 = sort direction (e.g., "ASC", "DESC")
```

**In your application code**, you'd:
1. Validate the sort field against a whitelist of allowed fields
2. Validate the direction is either ASC or DESC
3. Construct the query with the validated values

### Pagination

For large datasets, never return all rows at once.

```sql
SELECT u.*, row_to_jsonb(up.*) as profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
ORDER BY u.created_at DESC
LIMIT $1 OFFSET $2;
-- $1 = limit (e.g., 10, 20, 50)
-- $2 = offset (calculated from page number)
```

**Calculating offset from page:**
```
offset = (page_number - 1) * limit
```

**Example:**
- Page 1, limit 10 → offset 0
- Page 2, limit 10 → offset 10
- Page 3, limit 10 → offset 20

## 14. CRUD Operations

### CREATE (INSERT)

Add new data to the database.

```sql
INSERT INTO users (email, full_name, password_hash)
VALUES ($1, $2, $3)
RETURNING *;
-- Returns the created row
```

**Key points:**
- Specify columns explicitly (better for schema changes)
- Use parameterized queries for security
- `RETURNING *` gives you back the created data (useful for API responses)

### READ (SELECT)

Retrieve data from the database.

```sql
-- Get all users with profiles
SELECT u.*, row_to_jsonb(up.*) as profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
ORDER BY u.created_at DESC;

-- Get single user by ID
SELECT u.*, row_to_jsonb(up.*) as profile
FROM users u
LEFT JOIN user_profiles up ON u.id = up.user_id
WHERE u.id = $1;
```

### UPDATE

Modify existing data.

```sql
UPDATE user_profiles
SET 
  bio = $1,
  phone_number = $2
WHERE user_id = $3
RETURNING *;
-- Returns the updated row
```

**Partial updates:** In your application code, only include fields the user actually changed:
```python
# Pseudocode
update_fields = {}
if bio in request:
    update_fields['bio'] = request.bio
if phone_number in request:
    update_fields['phone_number'] = request.phone_number

# Only update provided fields
```

### DELETE

Remove data from the database.

```sql
DELETE FROM tasks
WHERE id = $1;
```

**Note:** With foreign key constraints using `ON DELETE CASCADE`, deleting a parent record automatically deletes related child records.

## 15. Triggers

### What Are Triggers?

Triggers are database functions that automatically execute in response to certain events (like INSERT, UPDATE, DELETE).

### Common Use Case: Auto-updating `updated_at`

Instead of manually updating `updated_at` in every UPDATE operation, use a trigger:

```sql
-- Create a function to update the timestamp
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Apply the trigger to a table
CREATE TRIGGER update_users_updated_at
  BEFORE UPDATE ON users
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();
```

**How it works:**
1. Before any UPDATE on the `users` table
2. For each row being updated
3. Set the `updated_at` field to the current timestamp
4. Continue with the update operation

**Benefits:**
- Automatic - no need to remember in application code
- Consistent - always updated, can't forget
- Centralized - change in one place affects all updates

**Apply to all tables:**
```sql
CREATE TRIGGER update_user_profiles_updated_at
  BEFORE UPDATE ON user_profiles
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER update_projects_updated_at
  BEFORE UPDATE ON projects
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();

-- And so on for each table...
```

## 16. Database Indexes

### What Is an Index?

Think of a book's index section. Instead of flipping through every page to find "Chapter 4", you look at the index which tells you "Chapter 4 starts on page 54" and jump directly there.

**In databases:** An index is a lookup table that maps field values to the physical location of rows on disk.

### The Problem Without Indexes

When you query without an index:
```
Query: "Find task with ID = 'abc-123'"

Database action:
1. Go to first task location on disk
2. Check: Is ID = 'abc-123'? No
3. Go to second task location
4. Check: Is ID = 'abc-123'? No
5. Repeat for ALL tasks until found
```

This is a **sequential scan** - very slow for large datasets!

### The Solution With Indexes

With an index on the `id` field:
```
Query: "Find task with ID = 'abc-123'"

Database action:
1. Look at the index table (stored sequentially, fast to scan)
2. Find: 'abc-123' is at location X on disk
3. Go directly to location X
4. Return the row
```

Much faster! The index provides **direct access** instead of sequential scanning.

### When to Create Indexes

Create indexes on fields used in:

1. **WHERE clauses** - filtering conditions
2. **JOIN conditions** - matching related tables
3. **ORDER BY clauses** - sorting operations

**Examples:**

```sql
-- Index for WHERE clause: finding user by email
CREATE INDEX idx_users_email ON users(email);

-- Index for ORDER BY: sorting users by creation date
CREATE INDEX idx_users_created_at ON users(created_at DESC);

-- Index for JOIN condition: fetching tasks by project
CREATE INDEX idx_tasks_project_id ON tasks(project_id);

-- Index for WHERE clause: filtering tasks by status
CREATE INDEX idx_tasks_status ON tasks(status);
```

### Index Ordering

Indexes can be ascending or descending:

```sql
-- For queries that sort descending
CREATE INDEX idx_users_created_at_desc ON users(created_at DESC);

-- For queries that sort ascending
CREATE INDEX idx_users_created_at_asc ON users(created_at ASC);
```

Match the index order to your query's `ORDER BY` for best performance.

### The Trade-off

Indexes aren't free - they have a cost:

**Write overhead:**
- Every INSERT must update the index
- Every UPDATE on indexed fields must update the index
- Every DELETE must update the index

**Storage:**
- Indexes take up disk space

**When to index:**
- Index frequently queried fields
- Don't index everything (write-heavy tables might suffer)
- Monitor performance and adjust

**Rule of thumb:** If a query is called frequently and involves WHERE, JOIN, or ORDER BY on a field, consider indexing it. You can always remove the index later if the overhead isn't worth it.

### Primary Keys Are Auto-Indexed

Good news: Primary keys are automatically indexed by the database. You don't need to manually create indexes on primary key fields.
