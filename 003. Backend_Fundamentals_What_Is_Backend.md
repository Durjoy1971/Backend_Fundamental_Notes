# Backend Fundamentals: What Is a Backend and Why Do We Need It?

## What Is a Backend?

Let's start with the traditional definition. A backend is a computer that listens for HTTP, WebSocket, gRPC, or any other type of request through an open port (typically port 80 or 443) that is accessible over the Internet. Clients or other frontends connect to it to send data or receive data depending on the kind of request.

We call it a "server" because it serves content. This content could be:
- Static files like images, JavaScript files, or HTML files
- JSON data
- It also accepts data if the client sends something

That's a fair definition of what a backend is and how it works. But to truly understand backends, you need a holistic view of the physical components and how they work behind the scenes.

## Request Flow: From Browser to Server

Let's trace the complete journey of a request from your browser to the server. I'll walk through each hop in the chain.

### The Complete Journey

**1. Browser**

Everything starts when you make a request from your browser. For example, you might visit `backend-demo.doxyz`.

**2. DNS Server**

The first thing that happens is DNS resolution. DNS (Domain Name System) is a massive topic, but here are the basics you need to know:

DNS has different types of records:
- **A records**: Point to a particular IP address
- **CNAME records**: Point to a particular domain name or subdomain

In our example, we have an A record for `backend-demo` that points to a specific IP address. Where does this IP come from? It's the public IP address of an AWS EC2 instance.

**3. AWS EC2 Instance**

Your request reaches the AWS server through the IP address that DNS resolved. This is where your backend server is actually running.

**4. Firewall (Security Group)**

Before the request reaches the actual computer (the EC2 instance), it passes through a firewall. AWS has a native firewall system called "security groups."

Security groups let you specify which ports you want to allow to be accessible over the Internet. In our example, we allow three different kinds of ports:
- SSH port: Used to log into the AWS instance via terminal for operations
- Port 443: For HTTPS traffic
- Port 80: For HTTP traffic

This is critical: if you don't allow ports 443 or 80, AWS will block the request right here, and it will never reach your server.

**5. Reverse Proxy (Nginx)**

After passing through the firewall, the request reaches your EC2 instance. Here, we use something called a reverse proxy. A reverse proxy is a server that sits in front of other servers, allowing you to manage different types of redirects or configurations from a centralized space instead of changing configs in every single server.

We use Nginx as our reverse proxy. Here's what the configuration does:

- **Certbot**: Automatically assigns SSL certificates (not something you need to worry about for this demo)
- **Port 80 listening**: Nginx listens on port 80 and redirects requests to port 443 (HTTPS)
- **Server name**: Defines which domain this config applies to (e.g., `backend-demo`)
- **Redirection**: Whatever request comes to this domain gets redirected to `localhost:3001`—the port our Node.js server is running on

**6. Application Server (Node.js)**

This is the final hop. Our Node.js server runs on `localhost:3001`. We use PM2 to manage our processes, and you can see two processes running: one for the frontend and one for the backend.

From the point of view of this EC2 instance, our server runs on `localhost`. We use Nginx and domain names to route requests to our local server over the Internet.

### Summary of the Request Flow

```
Browser → DNS Server → AWS Server → Firewall → EC2 Instance → Nginx → Localhost:3001 (Node.js Server)
```

### Local Development

When you're developing locally, you can just open your browser and go to `localhost:3000/users` and you'll see the same response. This is exactly what happens on the AWS instance as well—if you call the server directly on localhost, you get the same response.

Now you should have a fair idea of how a request looks and how a request travels over the Internet to reach a server.

## Why Do We Need Backends?

This is an excellent question. Let me give you a real-world example.

Imagine you're scrolling through your Instagram feed and you come across your friend's post. As usual, you like it—you click on the like button. On the other side, your friend gets a notification that you liked their post.

Between you clicking the like button and your friend getting the notification, what exactly happened?

This is where the concept of backend comes into play:

1. You click the like button
2. The app sends a request to the server
3. The server processes the request: it sees who the user is, finds your name or ID
4. The server persists and saves the data—the action of you liking the post—usually in a database
5. The server checks who the user is whose post you liked, finds that user's ID
6. The server triggers some action that sends a notification to that user
7. Your friend receives the notification on their phone

All these interactions that happened between you clicking the like button and your friend receiving the notification require a centralized computer—a server—that has all kinds of information about all users.

Think about it: your app is designed and customized according to your needs, your profile, and all the people you follow. Similarly, your friend's app is customized for them—they only receive notifications intended for them. But a server has to have all kinds of information about all users and all kinds of state.

### The Core Responsibility of a Backend

If you try to condense the responsibility and use of a backend to a single word, it would be this: **DATA**

- The need to fetch data
- The need to receive data
- The need to persist data somewhere
- Any and every kind of action that deals with data

That's what backends are fundamentally about.

## Why Not Do Everything in Frontend?

You might ask: "Why not just do everything on the frontend? The frontend is also a computer (your phone, laptop, etc.). Why not connect to the database here and do every kind of action that servers do? Since everything is distributed everywhere, wouldn't we have better performance?"

That's an excellent question. To understand why we can't use all those functionalities in the frontend, we need to see how frontends actually work behind the scenes.

## How Frontends Work

Let's look at a frontend demo—a Next.js application deployed on the same AWS instance as our backend.

The request flow is similar to what we saw with the backend:
- DNS resolves the domain to the IP address
- Request goes to AWS, passes through the firewall
- Reaches the EC2 instance

The difference is in the Nginx configuration. Instead of redirecting to port 3001 (the backend), we redirect to port 3000 (the Next.js frontend server).

The Next.js server serves:
- HTML file
- JavaScript files
- CSS files
- Images
- Fonts

Here's what happens in the browser:

1. The browser fetches the HTML file first
2. Then it fetches all the resources (JavaScript, images, fonts, CSS files) in separate requests
3. After fetching all the CSS, the browser paints the window—this is how you get styles, background colors, fonts, and button styles
4. After fetching all the JavaScript files, the browser "hydrates" the event listeners for buttons and interactions

For example, when you click a button and it redirects to another page, that happens because the browser fetched the JavaScript, added the event listeners, and that's how the button started working.

### Key Difference: Backend vs Frontend Execution

Here's the crucial difference:

- **Backend**: You send a request to the server, the server processes the request, and sends you the result. The actual processing happens on the server.
- **Frontend**: The server sends you code, but the browser runs the code. Whatever logic runs, it is run by the browser.

Whatever frontend logic you've written—whatever JavaScript you've written—is all fetched by the browser from your server and executed by the browser in your machine (the client's machine). The browser is your runtime.

## Browser Sandbox and Security Restrictions

Browser runtimes are often **sandbox environments**. This means they are isolated from:
- Your operating system
- System processes
- The file system

Everything is an isolated environment. The code can only access a limited amount of resources:
- The DOM (Document Object Model)
- Browser APIs like localStorage or cookies
- External APIs—but only if those external APIs have all the required headers

### CORS (Cross-Origin Resource Sharing)

We haven't covered CORS in depth yet (we will in a future video), but you can imagine CORS as a security policy of browsers that restricts JavaScript code from calling external APIs that are not on the same domain.

For example, if you have a frontend app at `frontend-demo.doxyz`, you can only fetch resources or call external APIs that are in the same domain. If you try to call a different domain, the browser will block the request because of the CORS policy.

There are ways to get around this through HTTP headers, which we'll explore later.

### Why Browser Sandboxing Makes Sense

If you think about it, all these sandboxing and security restrictions make perfect sense. What is a browser doing? It's fetching code from a remote server and executing it in the user's browser.

If browsers didn't isolate environments, that remote code could easily access data or files from the user's computer. Imagine you visit a website and have no idea what the source code is. If browsers didn't isolate environments, that code could access your file system, copy all your files and sensitive details, and send them to their servers.

That's a very scary scenario—which is why browsers have all these strict security policies.

## Why Can't We Write Backend Logic in Frontend?

Now, coming back to our original question: why can't we write backend logic in the frontend? There are four main reasons.

### 1. Security Restrictions

Browser security policies are extremely restrictive. Backends often need to access the underlying file system—whether it's to write to a log file or access environment variables. Browsers simply won't allow this.

For a backend server, this is a huge restriction. You need to be able to write logs, read configuration, and access system resources. Browsers block all of this.

### 2. External API Restrictions (CORS)

You cannot call external APIs whenever you want. The API must have all the appropriate CORS headers. Since we don't have control over all external APIs, this is a major deal breaker.

Backend servers often need to connect to other servers and fetch data from multiple places. Having this restriction would severely limit what a backend can do. We cannot have this limitation.

### 3. Database Access

Server runtimes have access to native database drivers—for example, `pg` for PostgreSQL or drivers for MongoDB. These drivers are written to work in environments that can:
- Handle socket connections
- Handle binary data
- Maintain persistent connections

Browsers cannot do these things.

Backend servers maintain a **connection pool**—a list of connections to the database server—so that they don't have to create and destroy connections again and again. Backend servers receive thousands and thousands of requests per second. If they had to do the connecting and destroying logic with each request, the database server would get overwhelmed and couldn't handle that load.

Drivers are written in a way that can maintain a list of connections. Browsers are not designed to maintain persistent connections to databases. Even if they were, each user would need to open their own database connection, which would overwhelm the database server with too many connections.

There's also no easy way to manage connection pooling or efficient query execution from browser environments.

### 4. Computing Power

Frontend applications run everywhere:
- Smartphones
- Desktops
- Laptops
- Any kind of environment you can imagine

Some devices have very limited resources—a computer with 256MB of RAM and a single-core processor, for example. The user might not have enough computing power to perform heavy business logic. Things would start to lag and sometimes break because of the load.

With a centralized backend server that serves a large number of clients, you can easily increase its memory and CPU whenever you want to deal with the load.

We could continue this list as long as we want, but this should give you a fair idea that keeping backend logic in the frontend is not a good idea—assuming you could do it in the first place.
