# Routing Fundamentals

## Introduction to Routing

In the previous discussion, we talked about different HTTP methods and their importance in HTTP semantics. HTTP methods describe the "what" of a request—your intent, your action. They express what you want to do on a particular resource: fetch data, add data, update data, or delete data.

The role of routing is to express the "where" of a request. Where do you want to send your intention? Which resource do you want to perform your action on? You need to tell the server where you want to go.

For example, you have a request with the method GET. Your intention is fetching data from the server, and your route path is `/users`. The server sends you an array of users. What you're saying is: "I want to fetch some data. My action is fetching, and from where? This resource—this route."

The combination of these two—your intention (action) and your route (the URL path, the address)—is what the server uses. It takes both, maps them to a particular handler or set of instructions, performs all the business logic and database operations needed, and returns the data.

To summarize: routing is basically mapping URL parameters to server-side logic. That's all there is to it.

## Static Routes

The first type of routing we'll look at is static routes. These are routes with constant paths that don't have any variable parameters.

**What makes a route static?**

A static route is constant—the path never changes. For example, `/api/books` will always be `/api/books`. Nothing changes in that string, and it always returns the same kind of data.

**Example:**

Consider these two requests:
- GET `/api/books` → returns an array of books
- POST `/api/books` → creates a new book and returns all books

In both cases, the route part is the same: `/api/books`. What's different is the HTTP method. In the server, these two things—the method and the route—act as a key that maps to a particular handler. The server first checks the method, then checks the route, and concatenates them to form unique routing logic.

- GET `/api/books` → one unique path
- POST `/api/books` → another unique path

These two will never clash because the methods differentiate between them. Routes like `/api/books` for GET and POST are called static routes because they have no variable parameters inside the route.

## Dynamic Routes (Path Parameters)

Now let's look at dynamic routes. These include variable parameters that change based on input.

**Path Parameter Terminology**

Dynamic parameters in routes are called path parameters (or route parameters). They go right after the forward slash and are part of the route itself—that's why they're called path parameters.

**Example:**

Consider this route: `/api/users/123`

Here, `123` is supposed to express the ID of the user. The application of this API endpoint is that we can fetch details of a particular user using the ID inside the route parameter. When we make this request, the server can extract this particular ID from the route path and perform operations like fetching the user from the database and returning the details.

**Server Route Matching**

To give you an idea of how this works in code, here's what the route matching part looks like in a server:

```
r.get('/api/users/:id')
```

The server is saying: if any request comes with the method GET and the route `/api/users/` followed by any string—this `:id` part means any kind of string that comes in this format—route it to this handler.

**Important convention:** This colon notation (`:id`) is an industry-wide standard. You'll find it in any server—Java, Python, Node.js, Go, Rust—it doesn't matter. The colon says "this is a dynamic parameter that the user will send in place of ID."

**How It Works**

When the request `/api/users/123` comes in:
1. It matches the GET method
2. It matches `/api`
3. It matches `/users`
4. The next part is a string (123—even though it looks like a number, in route parameters everything is converted to a string)
5. This gets inserted into the dynamic parameter slot we're calling `id`

**Human-Readable Design**

If you read the route `/api/users/123`, it's clearly readable. You can read it as: "GET me some data from `/api/user`—whose data? The user with ID 123." This is the whole idea of REST APIs—they provide a human-readable construct to routing.

This is called a dynamic route, and the dynamic part (the `:id`) is called a path parameter or route parameter.

## Query Parameters

We have two different types of parameters, which can be confusing:
- **Path parameters**: Dynamic parameters that are part of the route (after the forward slash)
- **Query parameters**: Key-value pairs after the question mark

**What Are Query Parameters?**

Consider this URL: `/api/search?query=some+value`

The route part is `/api/search`. This is what the server uses to match to a handler. The query part—the question mark, the key, and the value—is called a query parameter.

**Why Do We Need Query Parameters?**

In POST or PUT requests, we have a request body. We can use the body to send user-defined values to the server. But in GET requests (in REST API), GET requests don't have a body. If we want to send some value in a GET request, how do we do it?

We could use path parameters, but path parameters serve a particular purpose—they provide semantic expression. In the previous example, `/api/users/123` semantically means "fetch the details of a user whose ID is 123." That's the semantic expression of a path parameter.

But if we want to call `/api/search` and say the search value is some random value the user typed into an input box, sending it as a path parameter would look like `/api/search/some+value`. Technically this is possible, but it's very hard to maintain and defeats the whole purpose of REST API—providing semantic expressions to endpoints.

That's why we have query parameters. In query parameters, we can send a set of key-value pairs in a request. We typically use them with GET requests because we don't have a body to send data in.

**Common Use Case: Pagination**

Let's say we have an endpoint that fetches a list of books in paged format:
- `/api/books` → returns an object with:
  - `data`: an array of books
  - Metadata like `total`, `currentPage`, `totalPages`

The purpose of this is pagination. The server paginates the data and sends a chunk. Let's say the limit is 20 (a default limit). The server sends 20 books from the start and lets the client know:
- Total books: 100
- Current page: 1
- Total pages: 5 (100 ÷ 20)

The client can use this metadata to make subsequent requests according to its needs. How can it fetch the next page? It can do `/api/books?page=2`. By default, the server sends page 1—we didn't have to send any query parameter. But for the second request, we want to tell the server we want page 2. In a GET request, we send that information using query parameters.

**Other Common Uses**

Query parameters are also used for:
- Filtering by user-defined parameters
- Sorting (what order: ascending or descending)
- Any metadata about the request

All these kinds of information go inside query parameters in the form of key-value pairs.

## Nested Routes

Nested routes aren't really a type of routing—they're a practice you'll see everywhere. In REST APIs, for semantic expression, we often have to resort to nesting different types of resources. The nested route is typically the result of that.

**Example:**

Consider this route: `/api/users/123/posts/456`

Breaking it down:
- `/api/users`: the static part (this is itself a route that returns a list of all users)
- `/123`: the first dynamic path parameter (user ID)
- `/posts`: indicates we're fetching posts
- `/456`: the second dynamic path parameter (post ID)

**Semantic Meaning**

Why do we do this? Again, to express semantic meaning. What we're saying is:
1. We want to do a GET operation on `/api/users/123` → fetch the details of user with ID 123
2. The first part expresses that we're fetching information related to this user
3. The second part says we're fetching the posts of that user
4. Going one level deep, we're fetching a particular post with ID 456

**Different Levels of Nesting**

We can stop at different phases:
- `/api/users` → returns list of all users
- `/api/users/123` → returns information of user with ID 123 (one level of nesting)
- `/api/users/123/posts` → returns all posts of user 123 (expresses a different meaning)
- `/api/users/123/posts/456` → returns a particular post of user 123

Each level matches a different handler in the server and expresses a different meaning. We're fetching the posts of a user with ID 123, and specifically post 456.

**Why It's Useful**

Nested routing is pretty convenient and you'll see it getting used pretty much everywhere if the API has even a medium level of complexity. It allows us to express hierarchical relationships between resources in a clear, human-readable way.

## Route Versioning and Deprecation

Next up is an interesting concept: route versioning and deprecation.

**What Is Route Versioning?**

Consider these two requests:
- `/api/v1/products`
- `/api/v2/products`

This is called route versioning. It's a very common practice in API endpoints for servers with a REST API interface.

**Why Do We Use Versioning?**

Let's look at the responses to understand:
- `/api/v1/products` → returns `{ data: [{ id, name, price }] }`
- `/api/v2/products` → returns `{ data: [{ id, title, price }] }`

In v1, the field was `name`. In v2, it's `title`. This is a trivial example, but the use case is real.

**Real-World Scenario**

Let's say you have an API endpoint returning data in a particular format. Later, new requirements came in—you were earlier serving a web app, and in the future you're serving a React Native app, an Android app, or a Flutter app. For that new platform, you had to change the response format of your data.

**Options:**

1. **Change the whole route**: You could do `/api/new-products`—but this changes the entire route matching part.
2. **Add versioning**: You can say "in version 1 of our API, we were serving responses in this format. New requirements came in, and in version 2, we're serving in this format."

**Benefits of Versioning**

1. **Expresses intention clearly**: You have v1 response format and v2 response format clearly labeled.
2. **No need to change the whole route**: You don't have to do `/api/new-products`.
3. **Migration window**: You can eventually deprecate v1. Send a notice to frontend engineers that after v2 is released, v1 is deprecated. In the next release, all engineers can migrate to v2.

**Workflow**

1. Release v2 with the new format
2. Notify frontend engineers that v1 is deprecated
3. Engineers have a window to migrate endpoints to v2 format
4. Eventually, completely remove v1 and make v2 the new v1

This gives you a very stable and complete workflow to add new structure to your API endpoints and add breaking changes. Your engineers (the clients) have a window where they have the opportunity to migrate to the new structure.

## Catch-All Routes

Finally, we have catch-all routes.

**What Is a Catch-All Route?**

A catch-all route handles requests to routes that the server doesn't serve. For example, if we request `/api/v3/products` but the server doesn't have a v3 endpoint, what happens?

**Default Behavior Without Catch-All**

If you don't do catch-all handling, the server would just send a null response.

**With Catch-All**

What the server typically does is: after serving all the different routes and all the methods associated with those routes, at the end it has a catch-all handler. Whatever request reaches this part after going through all previous route matching algorithms gets mapped to this handler.

**Purpose**

The handler sends a user-friendly message: "This route you're requesting does not exist. Route not found."

Instead of just sending a null response (the default behavior), we send a user-friendly message to let the client know that we don't cater to that endpoint.

**Implementation**

This is usually done with something like `/*` or similar wildcard notation at the end of the route matching algorithm.

That's all catch-all route is about—providing a better user experience for unmatched routes.
